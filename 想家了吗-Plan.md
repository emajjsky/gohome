# 想家了吗 Plan

## 1. 文档目的
这份 Plan 用来定义 `想家了吗` 在当前代码仓库中的实施步骤、阶段划分、交付顺序和执行约束。

从现在开始，后续继续开发时，默认先看 `PRD`，再按本 `Plan` 执行。

## 2. 执行基线
执行时默认遵守以下文档顺序：

1. `想家了吗-PRD.md`
2. `想家了吗-Plan.md`
3. 当前代码实现

出现冲突时的处理顺序：
- 先看 PRD 是否已经定义
- 再看 Plan 是否已有执行顺序
- 最后才直接改代码

## 3. 当前仓库基线
当前项目基于以下目录和文件继续推进：
- 页面文件：`index.html`、`monitor.html`、`events.html`、`event_detail.html`、`companionship.html`、`privacy.html`
- 回忆模式页面：`memorial_home.html`、`digital_human.html`、`memory_gallery.html`、`voice_archive.html`
- 二级设置页：`connect.html`、`rules.html`
- 样式文件：`assets/styles/app.css`、`assets/styles/tailwind.css`
- 资源文件：`assets/images/*`、`assets/fonts/*`

## 4. 总体实施策略
当前不做推倒重来，而采用“在现有原型上持续收敛”的方式实施。

原则上按这 4 条推进：
- 先统一产品表达，再补功能细节
- 先修页面主链路，再修二级页
- 先做减法，再决定是否加功能
- 先保证全站一致，再做局部精修

## 5. 实施阶段
### 阶段 0：文档定版
目标：
- 形成可执行的 PRD 和 Plan
- 让后续修改不再只靠口头反馈

交付物：
- `想家了吗-PRD.md`
- `想家了吗-Plan.md`

完成标准：
- 产品目标清晰
- 技术栈清晰
- MVP 范围清晰
- 执行顺序清晰

### 阶段 1：主链路稳定
目标：
- 让普通模式主链路完整可用

涉及页面：
- `index.html`
- `monitor.html`
- `events.html`
- `companionship.html`
- `privacy.html`

关键动作：
- 首页负责触发
- 守护页负责确认
- 时间线负责证据
- 陪伴页负责关系变化
- 我的页负责设置开关和阈值

完成标准：
- 每个主页面职责单一明确
- 主页面不过度解释
- 底部导航统一

### 阶段 2：回忆模式稳定
目标：
- 让回忆模式形成独立完整链路

涉及页面：
- `memorial_home.html`
- `digital_human.html`
- `memory_gallery.html`
- `voice_archive.html`
- `privacy.html?memorial=on`

关键动作：
- 回忆模式首页负责推荐入口
- 平行世界负责情绪出口
- 记忆馆负责回忆片段
- 声音页负责声音触发
- 我的页负责开关与资料准备

完成标准：
- 普通模式与回忆模式可顺畅切换
- 平行世界像聊天，不像说明页
- 记忆内容按片段组织，不是素材堆叠

### 阶段 3：二级页统一
目标：
- 让二级页不掉队，不再像旧原型残留

涉及页面：
- `event_detail.html`
- `connect.html`
- `rules.html`

关键动作：
- 压缩顶部说明
- 保留底部导航
- 保持和主页面同一语言体系

完成标准：
- 二级页不再出现过量解释
- 二级页交互回路完整

### 阶段 4：视觉收口
目标：
- 做最后的统一和减法

重点检查：
- 是否仍有多余英文小标题
- 是否仍有情绪化标签乱用
- 是否仍有字顶边、溢出、贴框
- 是否仍有花哨底纹和重边框
- 是否仍有不必要说明段落

完成标准：
- 页面更像产品，不像原型
- 所有主视图都足够克制
- 所有文字都在安全边界内

## 6. MVP 实现步骤
以下步骤按真实执行顺序推进。

### 步骤 1：锁定技术栈
- 继续使用多页静态 HTML
- 继续使用 `Tailwind CSS + app.css`
- 继续使用本地字体、图标和图片
- 不引入新框架

### 步骤 2：先改主页面
- 首页
- 守护
- 事件
- 陪伴
- 我的

原因：
- 这些页面决定产品第一感

### 步骤 3：再改回忆模式
- 回忆模式首页
- 平行世界
- 记忆馆
- 声音

原因：
- 这是产品差异化的第二主链路

### 步骤 4：再收二级页
- 事件详情
- 接入设备
- 守护规则

原因：
- 二级页是统一感的最后短板

### 步骤 5：做全站减法
- 删多余标题
- 删多余文案
- 删多余标签
- 删多余说明卡
- 删不必要视觉装饰

## 7. 页面执行规则
### 7.1 主页面规则
- 不保留冗余顶部标题栏
- 进入页面先看到内容本体
- 主标题不允许过长导致溢出

### 7.2 二级页面规则
- 可以有页面标题，但不能有大段说明
- 必须保留底部导航或清晰返回路径

### 7.3 标签规则
- 标签只做分类
- 禁止使用“最想回家”“有点惦记”这类主观标签

### 7.4 文案规则
- 短句优先
- 动作优先
- 避免解释性长段落

### 7.5 样式规则
- 面板底纹克制
- 组件边框轻
- 卡片留白稳定
- 字号和行高要在安全边界内

## 8. 工程执行规则
### 8.1 修改前
- 先核对是否符合 PRD
- 先看当前页面职责是否清晰

### 8.2 修改中
- 优先做减法
- 单页修改完成后做诊断检查
- 新命名必须和现有全站命名一致

### 8.3 修改后
- 检查页面文案是否变多了
- 检查是否破坏导航一致性
- 检查是否有文字顶边、溢出、贴框

## 9. 当前阶段优先级
### P0
- 保持普通模式和回忆模式两条主链路稳定
- 继续压缩全站多余说明
- 统一剩余页面的导航、标题和标签规则

### P1
- 增加更细的交互细节
- 补更完整的回忆触发素材
- 做更完整的展示和讲述版本

## 10. 执行承诺
从本次开始，后续执行默认遵守以下方式：
- 先对照 `PRD`
- 再对照 `Plan`
- 再做代码修改

如果执行过程中出现新需求：
- 若只是细节修正，可直接按文档原则执行
- 若涉及产品结构变化，应先更新 PRD 或 Plan，再继续开发
