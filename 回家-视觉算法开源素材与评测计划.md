# 回家视觉算法开源素材与评测计划

更新时间：2026-07-04

## 1. 目标

视觉算法不能靠“看起来能框出来”判断是否可用，必须进入固定闭环：

```text
开源素材 / 自采样本
-> 标准目录
-> 离线评测脚本
-> 阈值和模型版本
-> 树莓派真实摄像头回归
-> 管理端展示
```

本计划先解决四个问题：

- 准确：误报和漏报要能量化。
- 流畅：不能因为模型推理导致视频预览卡死。
- 合理：报警必须有时序、阈值、连续帧和场景约束。
- 好看：展示必须来自真实检测结果，而不是纯动效假演示。

## 2. 第一批推荐素材

### 2.1 人形 / 人体框

首选：COCO / COCO-Pose

- 用途：人形框、人体关键点、坐姿 / 半身增强的基础回归。
- 优点：下载直接，标注成熟，适合先跑 YOLO person 和 pose。
- 缺点：不是老人家庭场景，不能直接证明产品准确率。
- 使用策略：先取 val split 中 `person` 和 keypoints 子集，建立人形检测 baseline。

备选：Open Images V7

- 用途：室内负样本、家具、电视、灯光、多人场景。
- 优点：类别大，适合补误报样本。
- 缺点：数据量很大，先抽样。

### 2.2 跌倒 / 姿态

首选：UR Fall Detection Dataset

- 用途：跌倒 vs 日常活动回归。
- 内容：30 段跌倒、40 段日常活动。
- 优点：体量小，适合先把规则跑通。
- 缺点：Kinect 场景和真实 IP 摄像头不完全一致。

第二阶段：NTU RGB+D / NTU RGB+D 120

- 用途：跌倒、坐下、站起、吃饭、喝水等动作。
- 优点：动作类别完整，有 RGB 和骨架数据。
- 缺点：需要申请，体量大。

### 2.3 火灾 / 烟雾

首选：D-Fire

- 用途：火焰 / 烟雾目标检测。
- 优点：YOLO 标注格式，适合训练或评测轻量模型。
- 缺点：主要是图片，还需要视频素材做时序验证。

视频补充：Smoke and Fire Detection Videos Dataset

- 用途：验证动态火焰、烟雾、连续帧确认。
- 内容：85 个带框标注视频。
- 注意：使用前确认供应方授权条款。

### 2.4 用餐 / 喝水 / 日常动作

首选：EatSense

- 用途：吃饭、喝水、手到嘴、上半身动作。
- 优点：和“老人坐着吃饭/喝水”的产品需求更接近。
- 缺点：体量大，且需要核对研究数据授权。

备选：Kinetics / UCF101

- 用途：动作识别工程链路和模型预训练参考。
- 缺点：不是家庭监控视角，不能作为最终产品准确率证明。

### 2.5 视频流压力 / 负样本

推荐：VIRAT

- 用途：多目标、长视频、背景变化、视频管线压力测试。
- 缺点：主要是户外或公共监控，不是老人居家场景。

## 3. 算法评测分层

### P0：必须先做

- 人形检测：COCO person subset。
- 姿态 / 骨架：COCO keypoints + 当前 RTMPose 输出。
- 跌倒：UR Fall。
- 火灾：D-Fire + Fire/Smoke video。
- 生活观察：用真实摄像头定时样本验证，不用公开数据强行替代。

### P1：补产品场景

- 吃饭 / 喝水：EatSense、NTU RGB+D。
- 夜间活动：自采夜间摄像头片段 + Open Images 低光负样本。
- 长时间静止 / 发呆：主要依赖真实摄像头长时间采样。

### P2：模型预训练和扩展

- Kinetics。
- UCF101。
- VIRAT。

## 5. 评测指标

每次调整算法或阈值都必须输出：

- `precision`：命中的样本里有多少是真的。
- `recall`：真实异常里有多少被找出来。
- `false_positive_rate`：正常样本误报比例。
- `false_negative_count`：漏报数量。
- `latency_ms_p50 / p95`：推理耗时。
- `fps_preview`：页面预览帧率。
- `cpu_percent / memory_mb / temperature`：Pi5 压力。

## 6. 近期执行顺序

1. 建立 `edge-agent/data/eval` 标准目录。
2. 先下载小规模可直接用的数据：
   - COCO val person/keypoints 子集。
   - UR Fall。
   - D-Fire。
   - Fire/Smoke video 小样本。
3. 写离线评测脚本：
   - `eval-person.py`
   - `eval-pose.py`
   - `eval-fall.py`
   - `eval-fire.py`
4. 每个算法先出第一份 baseline 报告。
5. 再决定是否训练模型：
   - 人形继续用 YOLO。
   - 姿态继续用 RTMPose。
   - 火灾需要引入专门 fire/smoke detector。
   - 吃饭 / 喝水先用姿态 + 时间段 + 区域规则，不急着训练大模型。
6. 最后把评测结果回灌到管理端展示。

## 7. 关键判断

- 公开数据集只能做底座，最终准确性必须靠真实家庭摄像头样本回归。
- 树莓派 Pi5 不适合同时跑多个重模型；必须做“低频分析 + 实时预览分离”。
- 正式告警不能只靠单帧结果，必须有连续帧、时序状态机和场景约束。
- 展示要惊艳，但展示内容必须来自真实检测输出：框、骨架、轨迹、证据截图、事件时间线。

## 10. 真实 UR Fall 小样本导入

烟测样本只能证明工具链可运行，不能证明跌倒算法准确。跌倒算法第一批真实回归使用 UR Fall Detection Dataset：

- 数据集官方页：`https://fenix.ur.edu.pl/~mkepski/ds/uf.html`
- 默认只下载 `fall-01/fall-02/adl-01/adl-02` 的 cam0 RGB 视频和逐帧特征 CSV。
- 逐帧标签按官方说明处理：`1` 是躺在地上，作为跌倒正样本；`-1` 是未躺倒，作为负样本；`0` 是正在跌倒的过渡姿态，默认跳过。
- 授权为 CC BY-NC-SA 4.0，当前只用于研发评测；商业使用前需要联系数据集作者确认。

本机或树莓派运行：

```bash
cd edge-agent
chmod +x scripts/run-ur-fall-eval.sh
./scripts/run-ur-fall-eval.sh
```

只导入不评测：

```bash
python scripts/import-ur-fall-sample.py
```

评测已有导入样本：

```bash
python scripts/eval-fall.py \
  --use-pose \
  --samples-dir data/eval/samples/fall/ur_fall \
  --manifest data/eval/samples/fall/ur_fall/manifest.jsonl \
  --detector-backend yolo \
  --yolo-model yolo11n.pt \
  --yolo-confidence 0.20 \
  --yolo-imgsz 960
```

输出位置：

```text
edge-agent/data/eval/samples/fall/ur_fall/
edge-agent/data/eval/reports/
```

## 9. manifest.jsonl 格式

没有标签也能跑，但只能输出扫描报告；有标签才能计算准确率、召回率和误报率。

每行一个 JSON：

```json
{"file":"positive_001.jpg","person_present":true}
{"file":"negative_001.jpg","person_present":false}
```

跌倒：

```json
{"file":"fall_001.mp4","fall":true}
{"file":"sit_down_001.mp4","fall":false}
```

火灾：

```json
{"file":"fire_001.mp4","fire_event":true}
{"file":"warm_light_001.mp4","fire_event":false}
```

用法：

```bash
python scripts/eval-fire.py \
  --samples-dir data/eval/samples/fire \
  --manifest data/eval/samples/fire/manifest.jsonl \
  --target event
```

## 12. 2026-07-05 产品化检测逻辑补充

视觉算法的产品逻辑按三层处理，不再把模型单帧输出直接等同于用户告警。

### 12.1 三层输出

第一层：实时预览

- 目标是让用户和开发者看到算法在当前画面上识别到了什么。
- 可以展示人框、骨架、热区、置信度、模型版本和单帧候选。
- 这一层允许灵敏一些，但必须清楚标注为预览或候选。

第二层：规则确认

- 目标是把单帧候选变成可解释的事件候选。
- 必须结合连续帧、持续时间、同一目标、画面变化、恢复状态、区域约束和频控。
- 跌倒、火灾这类高危事件不能只靠单帧命中直接上报。

第三层：正式事件

- 目标是生成 App 可见、可上传、可追踪的事件记录。
- 必须包含事件类型、级别、摄像头、时间、截图证据、模型版本、规则阈值、观测值和可解释原因。
- 正式事件进入上传队列，由 App 服务器保存和分发。

### 12.2 跌倒当前判定逻辑

跌倒检测由 YOLO 人框、RTMPose 骨架和规则状态机共同决定：

- YOLO 负责人体框、贴地人体候选和多人片段。
- RTMPose 负责骨架关键点和姿态跌倒分数。
- 规则引擎负责状态流转：
  - `clear`
  - `visual_only`
  - `suspect`
  - `confirming`
  - `confirmed`
  - `recovered`
- 正式跌倒事件需要连续确认，不允许单帧直接生成高危事件。

当前评测命令：

```bash
cd edge-agent
./scripts/run-ur-fall-eval.sh --no-download
```

当前小样本结果：

- UR Fall 默认小样本 20 条。
- 正样本 8 条，负样本 12 条。
- 当前候选评测为 `TP 8 / FP 0 / TN 12 / FN 0`。
- 树莓派 CPU 上 RTMPose 首帧加载和逐帧推理较慢，产品页必须保证视频链路不被推理阻塞。

### 12.3 下一批样本

为了避免“只在当前画面好看”，下一批必须补负样本和难例：

- 坐下、起身、弯腰、捡东西。
- 躺沙发、床上翻身、地毯上坐卧。
- 多人遮挡、半身入画、低角度摄像头。
- 夜间低光、屏幕强光、阳光暖色、灯光反射。
- 摄像头画面卡顿、黑屏、低码率、严重噪声。

这些样本进入评测集后，跌倒、人体、骨架、火灾、生活观察都必须用同一套报告输出 precision、recall、false positive rate 和延迟。
