# 想家了吗 PRD

更新时间：2026-06-28

## 1. 文档定位

这份 PRD 定义 `想家了吗` 的商业化产品方向、系统边界、核心模块、数据模型、视觉算法路线和阶段验收标准。

本项目不是一个单纯的本机验证项目，也不是一个只跑在本机的最小功能样品。当前这台 M4 / 24GB Mac 作为第一阶段本地算力服务，用来证明边缘端视频接入、视觉检测、规则判断、事件生成、前端展示和通知链路可行。它不是最终商业硬件，但必须按未来边缘盒的运行方式来设计。

商业化目标产品是：

> 面向独居老人和异地子女的“居家安全与陪伴守护系统”，通过家庭边缘硬件、云端服务、视觉算法和用户端 App，持续感知老人居家状态，在异常发生时把可理解、可确认、可追踪的提醒推送给家属。

## 2. 产品愿景

`想家了吗` 的核心不是“让子女看监控”，而是“把老人居家状态转化成可行动的牵挂提醒”。

产品要解决三个问题：

- 子女无法长期知道老人独自在家是否安全。
- 普通摄像头只能直播或录像，不能把视频转化为风险事件。
- 老人安全提醒需要在隐私、误报、实时性和可解释之间取得平衡。

商业化后的产品应该做到：

- 老人家里放置一个边缘守护盒或预装服务的终端。
- 终端连接本地摄像头、毫米波雷达或其他传感器。
- 视频和传感器数据优先在本地处理，减少隐私暴露。
- 云端只接收事件、结构化结果、必要截图或短视频片段。
- 子女端 App 接收提醒、查看证据、处理事件、配置规则。
- 平台端支持设备运营、模型版本、告警质量和售后诊断。

## 3. 商业化形态

### 3.1 目标客户

第一阶段目标客户：

- 独居老人家庭
- 子女不在同城的家庭
- 有跌倒、走失、慢病风险的老人家庭

第二阶段可扩展客户：

- 社区养老服务站
- 居家养老上门服务机构
- 养老院和日间照料中心
- 保险、健康管理和银发服务渠道

### 3.2 产品包

商业化产品可以拆成三类：

- 家庭版：一台边缘盒 + 1 到 3 路摄像头 + 子女端 App。
- 社区版：多老人家庭接入 + 社区坐席管理台 + 家属通知。
- 机构版：多房间多摄像头 + 分级告警 + 值班工作台 + 运营报表。

### 3.3 收费模式

可选商业模式：

- 硬件一次性售卖 + 云服务订阅。
- 设备租赁 + 月服务费。
- 社区或机构按老人数量、设备数量或告警服务等级收费。
- 高级算法能力、短信电话通知、人工坐席作为增值服务。

### 3.4 产品边界

必须明确：

- 本产品是居家安全辅助提醒工具，不是医疗诊断设备。
- 跌倒、异常活动、无人活动等结果均为算法候选，需要人工确认。
- 产品价值在于尽早提醒和降低遗漏，不承诺 100% 识别所有事故。
- 隐私设计优先于直播能力，默认不鼓励长期打开视频监控。

### 3.5 商业版本定义

为了避免把阶段验证误认为正式产品，版本定义如下：

| 版本 | 定位 | 目标 | 典型形态 |
| --- | --- | --- | --- |
| V0 本机算力验证版 | 技术可行性验证 | 跑通 RTSP 摄像头、检测、事件、Web 展示、临时通知 | 当前 M4 / 24GB Mac + `edge-agent` + Web 原型 |
| V1 家庭试点版 | 小规模真实家庭试点 | 验证边缘盒稳定性、RTSP 兼容、告警质量、家属使用流程 | 边缘盒 + 云端最小平台 + H5/App 原型 |
| V2 商业家庭版 | 可销售产品 | 支持设备绑定、正式推送、售后诊断、订阅服务 | 量产边缘盒 + 正式 App + 云端平台 |
| V3 社区 / 机构版 | 多设备运营 | 支持多老人、多坐席、告警升级、运营报表 | 设备集群 + 运营后台 + 服务套餐 |

当前开发处于 V0 到 V1 之间。所有本机代码都应该服务于 V1/V2 的架构演进，不能只为了本地验证写死。

### 3.6 当前实施策略

当前阶段采用“强本机算力先行，硬件盒子后置验证”的策略。

- 第一阶段用当前 M4 / 24GB Mac 跑 `edge-agent`，它承担本地算力服务、摄像头拉流、YOLO 检测、规则判断、事件落库和管理台服务。
- 当前优先接入局域网 RTSP 摄像头，不再把本机摄像头 `local:0` 作为主线路。
- 树莓派或其他小盒子硬件可以开始采购，但不立刻作为主开发环境。
- 硬件板子的第一任务是验证部署、功耗、散热、开机自启和 24 小时稳定性，不承担当前产品闭环的主要开发风险。
- 在连接管理、规则配置、事件链路、通知链路和本地服务稳定前，不把主要开发迁移到树莓派。

这样做的原因：

- 当前 Mac 算力充足，可以更快验证 YOLO、规则引擎、事件模型和用户体验。
- 树莓派适合做低功耗硬件验证，但会引入系统依赖、性能、散热、模型量化和部署复杂度。
- 产品核心风险目前不在“有没有板子”，而在“摄像头能否稳定接入、算法结果能否解释、告警能否被家属正确处理”。

## 4. 用户角色

### 4.1 老人

老人不是主要操作用户，但必须被保护。

关注点：

- 不被复杂设备打扰。
- 不被持续直播侵犯隐私。
- 异常时能尽快被家人知道。
- 设备断网或断电时不要悄悄失效。

### 4.2 子女 / 家属

子女是主要 App 用户。

核心需求：

- 知道家里现在是否平稳。
- 收到异常提醒并快速判断严重程度。
- 查看必要截图或短视频证据。
- 一键确认、误报、联系老人或通知其他家属。
- 调整提醒规则，不被无效告警打扰。

### 4.3 安装和运维人员

核心需求：

- 快速完成设备绑定、摄像头接入和网络配置。
- 看懂摄像头是否在线、算法是否运行、通知是否可达。
- 远程诊断设备状态、日志和版本。
- 安全地升级模型和服务。

### 4.4 平台运营人员

核心需求：

- 查看设备在线率、告警量、误报率。
- 管理用户、设备、套餐、模型版本。
- 处理异常工单和售后问题。
- 监控服务质量和推送送达率。

## 5. 产品级系统架构

商业化系统由五层组成：

```text
用户端 App / Web
-> 云端业务平台
-> 设备连接与 API 网关
-> 家庭边缘硬件 edge-agent
-> 摄像头 / 传感器 / 本地视觉模型
```

### 5.1 用户端

包含：

- iOS App
- Android App
- Web H5 或 WebView 页面
- 家属协作入口

职责：

- 登录和家庭空间管理。
- 绑定守护设备。
- 查看设备和摄像头状态。
- 查看实时摘要、告警列表、告警详情。
- 接收推送通知。
- 配置看护规则。
- 邀请其他家属。
- 处理告警和反馈误报。

### 5.2 云端业务平台

包含：

- 用户与家庭空间服务
- 设备注册和绑定服务
- 摄像头配置服务
- 规则配置服务
- 事件服务
- 媒体服务
- 通知服务
- 家属协作服务
- 运营管理后台
- API 管理和鉴权

职责：

- 不直接拉取老人家摄像头 RTSP。
- 维护用户、设备、规则、事件和通知状态。
- 接收 edge-agent 上传的结构化结果。
- 向 App 提供统一 API。
- 负责推送、短信、电话等外部通知。
- 管理模型版本、设备版本和远程配置。

### 5.3 边缘硬件端

第一版用当前 M4 / 24GB Mac 模拟商业化边缘盒，商业化后再迁移到独立边缘硬件。

职责：

- 开机自启。
- 主动连接云端。
- 连接局域网摄像头。
- 本地拉流、抽帧、缓存。
- 执行基础视觉检测和规则判断。
- 上传事件、截图、短视频和健康心跳。
- 接收云端配置和模型更新。
- 网络异常时本地缓存，恢复后补传。

### 5.4 视觉算法层

包含：

- 视频接入
- 抽帧
- 图像质量检测
- YOLO 人形检测
- 多帧跟踪
- 姿态或动作模型
- 规则引擎
- 事件融合

算法层不能只等同于“跑一个 YOLO”。YOLO 是目标检测基础能力，产品级异常判断需要把模型结果、时间窗口、空间区域、摄像头状态和业务规则组合起来。

### 5.5 API 管理层

包含：

- REST API
- WebSocket / MQTT 设备通道
- OpenAPI 文档
- API 版本管理
- 设备鉴权
- 用户鉴权
- 权限控制
- 审计日志
- 限流和幂等

### 5.6 系统职责矩阵

| 子系统 | 拥有的数据 | 执行的逻辑 | 不应该做的事 |
| --- | --- | --- | --- |
| 用户端 App / H5 | 用户可见状态、表单输入、会话态 | 展示、交互、告警处理动作 | 不直接连 RTSP，不跑视觉模型 |
| 云端业务平台 | 用户、家庭、设备、规则、事件、通知、媒体索引 | 鉴权、状态机、通知编排、配置下发 | 不直接拉家庭摄像头，不保存无授权原始视频 |
| API 网关 / 设备通道 | token、连接状态、请求日志 | 鉴权、限流、路由、审计、协议适配 | 不承载业务判断，不保存大量媒体 |
| 边缘端 edge-agent | 本地配置、帧缓存、检测结果、待同步队列 | 拉流、抽帧、模型推理、规则执行、断网缓存 | 不管理用户账号，不暴露公网摄像头 |
| 视觉算法层 | 模型、检测结果、置信度、轨迹 | 图像质量、人形、姿态、时序分析 | 不直接决定通知家属，不替代业务规则 |
| 运营后台 | 设备质量、告警质量、用户反馈、工单 | 售后诊断、模型灰度、数据报表 | 不面向老人日常使用，不绕过权限查看隐私内容 |

### 5.7 端到端数据流

商业化数据流必须这样拆：

```text
摄像头原始视频
-> 边缘端帧缓存
-> 视觉模型 DetectionResult
-> 规则引擎 RuleEvaluation
-> 候选事件 EventCandidate
-> 云端事件 Event
-> 通知 Notification
-> 用户处理 UserAction
-> 反馈 Feedback
-> 模型和规则优化
```

关键约束：

- 原始视频默认不离开老人家。
- 上传云端的是事件、结构化检测结果、必要截图或短视频。
- App 展示的是产品化提醒，不展示算法内部对象。
- 运营后台看到的是质量指标和诊断信息，不默认暴露隐私媒体。

## 6. 模块职责

### 6.1 用户端 App

P0 商业功能：

- 登录 / 注册
- 创建家庭空间
- 绑定守护盒
- 设备在线状态
- 摄像头列表
- 最新状态摘要
- 告警列表
- 告警详情
- 截图或短视频查看
- 告警处理：已确认、误报、稍后提醒
- 看护规则开关
- 推送通知

P1 功能：

- 邀请家属
- 告警升级策略
- 联系老人快捷入口
- 家属处理记录
- 远程实时预览
- 设备安装向导

### 6.2 产品 Web 前端

当前 Web 原型承担两个职责：

- 阶段 0：作为 App 前的产品验证界面。
- 阶段 1：作为 H5 / WebView / 管理辅助页面复用。

商业化后需要拆分：

- 用户端 H5：给家属看状态、告警和规则。
- 安装调试页：给安装人员接入摄像头。
- 运营后台：给平台运营和售后诊断。

### 6.3 云端后端

核心服务：

- `identity-service`：用户、家庭、角色和权限。
- `device-service`：设备注册、绑定、心跳、版本。
- `camera-service`：摄像头配置、状态和接入能力。
- `rule-service`：规则模板、家庭规则、下发版本。
- `event-service`：事件入库、查询、状态流转。
- `media-service`：截图、短视频、留存和访问授权。
- `notification-service`：App 推送、短信、电话、机器人通知。
- `model-service`：模型版本、能力开关、灰度策略。
- `ops-service`：运营报表、设备诊断、审计日志。

### 6.4 edge-agent

商业化 edge-agent 不是单个脚本，而是边缘设备运行时。

模块：

- `device-agent`：设备身份、心跳、云端连接、远程配置。
- `camera-agent`：ONVIF 扫描、RTSP 拉流、断线恢复。
- `stream-agent`：FFmpeg / OpenCV 拉流、帧缓存、短视频缓存。
- `vision-agent`：YOLO、图像质量、姿态、跟踪。
- `rule-agent`：时间窗、区域、阈值、家庭规则。
- `event-agent`：事件生成、去重、节流、补传。
- `media-agent`：截图、短视频、脱敏、水印和本地留存。
- `update-agent`：服务升级、模型升级、回滚。
- `watchdog`：进程保活、异常重启、磁盘清理。

### 6.5 硬件端

商用品类：

- 当前：M4 / 24GB Mac，作为阶段 0 本地算力服务。
- 早期试点：Mac mini / N100 小主机 / 工控机。
- 低功耗验证：Raspberry Pi 5，优先验证部署、功耗、散热、网络和 24 小时稳定性。
- AI 加速验证：Raspberry Pi AI HAT+ 或同类 NPU 加速模块，在需要树莓派本地高频模型推理时再评估。
- 产品化：无风扇低功耗边缘盒。
- 可选：带 NPU 的国产或 ARM 边缘设备。

硬件要求：

- 长期 24 小时运行。
- 支持有线网络，Wi-Fi 作为备选。
- 支持至少 1 到 4 路 1080p 摄像头抽帧检测。
- 本地存储事件截图和短视频缓存。
- 断电恢复后自动启动。
- 支持远程升级和日志采集。
- 具备设备唯一身份和安全密钥。

## 7. 视觉模型与算法方案

### 7.1 视觉能力分层

视觉能力分为四层：

| 层级 | 能力 | 技术方案 | 产品用途 |
| --- | --- | --- | --- |
| L0 图像质量 | 黑屏、遮挡、模糊、过暗 | OpenCV 亮度、对比度、清晰度 | 保障摄像头可用 |
| L1 目标检测 | 人、宠物、轮椅、床、沙发 | YOLOv8/YOLOv10/YOLO11 系列 | 判断画面中是否有人 |
| L2 时序理解 | 停留、离开、长时间无人、活动量下降 | 多帧跟踪 + 状态机 | 降低单帧误判 |
| L3 行为风险 | 疑似跌倒、异常卧地、夜间活动 | 姿态模型 + 目标检测 + 时序规则 | 生成高优先级告警 |

### 7.2 YOLO 选型

阶段 0 可以使用 `YOLOv8n`，原因是：

- 模型小，适合当前 Mac 本地算力验证和未来低功耗硬件验证。
- 能快速得到人形检测框、置信度和人数。
- 接入成本低，适合跑通链路。

产品化阶段需要评估：

- `YOLOv8n`：速度快，适合低算力边缘盒，精度较低。
- `YOLOv8s` / `YOLOv8m`：精度更高，硬件成本上升。
- YOLO Pose：可做人关键点，为跌倒候选提供更多依据。
- 轻量姿态模型：用于跌倒、坐卧、长时间卧地。
- 模型量化：ONNX / NCNN / OpenVINO / CoreML / TensorRT，视硬件而定。

### 7.3 YOLO 在本产品里的角色

YOLO 只负责回答：

- 画面里是否有人。
- 人在哪里。
- 有几个人。
- 检测框大小、位置、置信度是多少。

YOLO 不直接回答：

- 老人是否安全。
- 是否一定跌倒。
- 是否需要通知家属。

产品级判断必须由规则引擎和时序逻辑完成：

```text
YOLO 检测结果
+ 最近 N 秒轨迹
+ 摄像头所在房间
+ 家庭规则
+ 时间段
+ 历史活动基线
+ 事件去重和置信度
= 产品级提醒
```

### 7.4 核心检测算法

P0 到 P1 的检测逻辑：

- 摄像头离线：RTSP 拉流失败、连续抓帧失败、心跳超时。
- 黑屏 / 遮挡：亮度、对比度、边缘信息和连续帧判断。
- 长时间无画面变化：帧差低于阈值并持续超过规则时间。
- 长时间无人：YOLO 连续 N 帧无 person 且持续超过规则时间。
- 异常卧地候选：人框横纵比、位置、持续时间和地面区域联合判断。
- 疑似跌倒候选：快速姿态变化、卧地持续、低位区域、无后续起身。
- 夜间异常活动：时间段 + 人形出现 + 活动轨迹。
- 区域停留：预设区域 + 停留时长 + 摄像头房间语义。

### 7.5 误报治理

商业化能否成立，关键不是“模型能不能识别一次”，而是长期误报是否可控。

必须建立：

- 事件置信度。
- 事件合并和节流。
- 用户误报反馈。
- 家庭级阈值调节。
- 摄像头位置校准。
- 模型版本灰度。
- 告警复盘数据集。
- 可解释证据：截图、时间线、检测框、规则命中原因。

## 8. 数据与逻辑分层

### 8.1 分层原则

商业化架构必须把数据和逻辑拆开：

- 原始视频：默认留在边缘端，不上传云端。
- 截图和短视频：只在事件触发时上传，且有留存策略。
- 检测结果：结构化上传，用于 App 展示、统计和误报治理。
- 业务规则：云端配置，边缘端执行。
- 用户状态：云端保存，App 查询。
- 设备运行状态：边缘端上报，云端用于诊断。

### 8.2 数据对象

核心对象：

- `User`：用户账号。
- `Household`：家庭空间。
- `Member`：家庭成员和角色。
- `ElderProfile`：老人基础信息、风险偏好和联系信息。
- `Device`：边缘盒设备。
- `Camera`：摄像头配置和状态。
- `RuleSet`：看护规则集合。
- `DetectionResult`：单帧或多帧算法结果。
- `Event`：产品级告警事件。
- `MediaAsset`：截图、短视频、缩略图。
- `Notification`：推送、短信、电话等通知记录。
- `Feedback`：已处理、误报、严重程度反馈。
- `AuditLog`：配置、查看、处理的审计记录。

### 8.3 事件生成链路

事件链路：

```text
Frame
-> DetectionResult
-> RuleEvaluation
-> EventCandidate
-> Event
-> Notification
-> UserAction
-> Feedback
```

说明：

- `DetectionResult` 是算法事实。
- `RuleEvaluation` 是规则命中过程。
- `EventCandidate` 是候选事件，可被合并或抑制。
- `Event` 是用户端可见事件。
- `Feedback` 用于后续规则和模型优化。

### 8.4 数据适配

前端、App、云端和边缘端不能直接共享内部结构，必须通过适配层：

- 边缘端内部：以帧、检测结果、规则状态为主。
- 云端内部：以家庭、设备、事件、媒体、通知为主。
- 用户端接口：以“家里状态、提醒、可执行动作”为主。
- 运营后台接口：以设备质量、告警质量、模型效果为主。

## 9. API 管理

API 必须从第一阶段就按产品化方式设计。

### 9.1 API 分类

- 用户端 API：App / H5 使用。
- 设备端 API：edge-agent 与云端通信。
- 管理端 API：运营后台和安装调试使用。
- 内部 API：模型、事件、通知、媒体服务之间调用。

### 9.2 用户端 API

核心领域：

- 登录和身份。
- 家庭空间。
- 设备绑定。
- 摄像头列表。
- 首页摘要。
- 告警列表。
- 告警详情。
- 告警处理。
- 看护规则。
- 家属协作。
- 媒体访问授权。

### 9.3 设备端 API

核心领域：

- 设备注册。
- 设备鉴权。
- 心跳上报。
- 配置拉取。
- 摄像头状态上报。
- 事件上报。
- 媒体上传。
- 日志上报。
- 版本更新。
- 模型更新。

通信策略：

- 设备主动连云端，避免家庭公网 IP 和端口映射。
- 配置和事件使用 HTTPS。
- 实时控制和在线状态可使用 WebSocket 或 MQTT。
- 媒体上传使用预签名 URL 或对象存储凭证。

### 9.4 API 治理

必须支持：

- `/api/v1` 版本命名。
- OpenAPI 文档。
- 设备 token 和用户 JWT 分离。
- 家庭角色权限控制。
- 请求幂等键。
- 事件去重键。
- 限流。
- 审计日志。
- 兼容旧设备版本。

### 9.5 API 领域草案

商业化 API 至少需要这些领域：

| 领域 | 面向对象 | 示例能力 |
| --- | --- | --- |
| Identity API | 用户端 | 登录、刷新 token、退出登录 |
| Household API | 用户端 | 创建家庭、成员列表、角色权限 |
| Device API | 用户端 / 设备端 / 管理端 | 设备绑定、解绑、心跳、版本、在线状态 |
| Camera API | 用户端 / 设备端 / 管理端 | 摄像头配置、状态上报、测试结果 |
| Rule API | 用户端 / 设备端 | 规则读取、保存、版本下发、设备确认 |
| Event API | 用户端 / 设备端 | 事件上报、列表、详情、确认、误报 |
| Media API | 用户端 / 设备端 | 上传截图、上传短视频、授权访问 |
| Notification API | 用户端 / 云端内部 | 推送设置、通知记录、送达状态 |
| Model API | 设备端 / 管理端 | 模型版本、灰度策略、能力开关 |
| Ops API | 管理端 | 设备诊断、告警质量、工单、审计 |

API 返回给用户端时，要隐藏技术细节，把字段适配成“房间、时间、提醒原因、建议动作、证据材料”。

## 10. 前后端职责拆分

### 10.1 用户端前端

前端只负责：

- 展示状态和提醒。
- 发起用户动作。
- 展示截图、短视频和处理建议。
- 管理规则配置表单。

前端不负责：

- 直接连接 RTSP。
- 执行视觉算法。
- 判断跌倒是否成立。
- 持久化核心业务状态。

### 10.2 云端后端

云端负责：

- 用户、家庭、设备和权限。
- 规则版本和下发。
- 事件状态机。
- 通知编排。
- 媒体访问授权。
- 多端同步。
- 运营后台和审计。

### 10.3 边缘端

边缘端负责：

- 摄像头连接。
- 本地视觉计算。
- 本地规则执行。
- 本地缓存和断点续传。
- 与云端保持设备通道。

## 11. 核心用户流程

### 11.1 设备安装

```text
安装人员通电边缘盒
-> 盒子联网
-> App 扫码绑定设备
-> edge-agent 上报设备在线
-> 添加或扫描摄像头
-> 测试画面
-> 配置房间和规则
-> 完成安装
```

### 11.2 日常守护

```text
edge-agent 本地抽帧
-> 视觉模型生成检测结果
-> 规则引擎判断是否异常
-> 本地保存截图 / 短视频
-> 上报事件到云端
-> 云端推送 App
-> 家属查看并处理
```

### 11.3 告警处理

```text
家属收到推送
-> 打开告警详情
-> 查看截图 / 短视频 / 命中原因
-> 选择已确认 / 误报 / 稍后提醒
-> 可联系老人或通知其他家属
-> 处理结果回写云端
-> 反馈进入规则和模型优化闭环
```

## 12. 阶段定义

### 12.1 阶段 0：本机验证

目标：

- 用当前 M4 / 24GB Mac 证明视频接入、检测、规则、事件、Web 展示和通知链路可行。

当前 `edge-agent` 属于阶段 0，不代表最终产品架构。

阶段 0 的主摄像头是局域网 RTSP 摄像头，用户端页面和管理台都应围绕真实摄像头配置、测试、规则、告警和处理流程展开。

### 12.2 阶段 1：产品化边缘端

目标：

- 把 `edge-agent` 从开发服务升级为可部署的边缘运行时。

关键点：

- 设备身份。
- 配置文件。
- 开机自启。
- Watchdog。
- 本地数据清理。
- 日志。
- 版本升级。
- RTSP 稳定性。

### 12.3 阶段 2：云端平台和 App

目标：

- 建立可商业化的用户、设备、事件和通知平台。

关键点：

- 用户账号。
- 家庭空间。
- 设备绑定。
- 云端事件中心。
- App 推送。
- 规则远程下发。

### 12.4 阶段 3：视觉模型产品化

目标：

- 让算法从“能跑”变成“可控、可解释、可运营”。

关键点：

- YOLO 检测框展示。
- 多帧跟踪。
- 姿态模型。
- 误报反馈。
- 模型版本管理。
- 场景数据集。
- 模型灰度和回滚。

### 12.5 阶段 4：硬件试点

目标：

- 在真实老人家庭中验证稳定性。

关键点：

- 24 小时运行。
- 断网恢复。
- 摄像头兼容性。
- 误报率。
- 推送送达率。
- 安装难度。
- 家属使用频率。

硬件试点启动条件：

- 当前 Mac 本地链路已经稳定运行。
- `connect.html` 可以完成真实摄像头添加和测试。
- `rules.html` 可以完成真实规则配置。
- 事件链路已拆成检测结果、规则评估、事件候选和用户事件。
- 至少一个真实手机通知通道已跑通。
- `edge-agent` 已具备开机自启、日志、配置文件和基础 watchdog 方案。

### 12.6 阶段 5：商业化运营

目标：

- 形成可销售、可交付、可售后的产品。

关键点：

- 设备交付流程。
- 售后诊断平台。
- 服务套餐。
- SLA。
- 隐私协议。
- 数据留存策略。

## 14. 当前明确不做

阶段 0 不做：

- 完整原生 App。
- 完整云端账号体系。
- 公网摄像头直播。
- 医疗级跌倒检测承诺。
- 大规模硬件适配。
- 复杂家庭协作。

商业化阶段也不做：

- 未经用户授权的持续云端录像。
- 把老人状态直接判定为医疗结论。
- 让 App 直接暴露或保存摄像头 RTSP 密码。
- 绕过边缘端把家庭摄像头接入公网。
