# 回家 PRD

更新时间：2026-07-09

## 1. 文档定位

这份 PRD 定义 `回家` 的商业化产品方向、系统边界、核心模块、数据模型、视觉算法路线和阶段验收标准。

### 1.1 文档对齐规则

为了避免后续实现走样，三份文档的关系固定如下：

- `PRD` 负责定义产品真相：
  - 产品目标是什么
  - 哪些能力属于正式产品
  - 阶段边界和完成标准是什么
- `Plan` 只能从 `PRD` 往下拆，不允许脱离 `PRD` 单独扩张新方向。
- `Implement` 只能记录“当前真实做到哪里”，不允许把原型能力写成正式产品已完成。

任何新能力进入开发前，都要先回答三个问题：

1. 这件事在产品里到底是什么能力。
2. 它属于哪个阶段，不属于哪个阶段。
3. 它在 PRD、Plan、Implement 三份文档里的表述是否一致。

本项目不是一个单纯的本机验证项目，也不是一个只跑在本机的最小功能样品。Mac 已经证明边缘端视频接入、视觉检测、规则判断、事件生成、前端展示和通知链路具备技术基线；当前主验证对象切到树莓派盒子，用来证明真实家庭硬件能独立联网、拉流、检测、预览、诊断和报警。Mac 不是最终商业硬件，后续所有能力必须按未来边缘盒的运行方式来设计。

商业化目标产品是：

> 面向独居老人和异地子女的“居家安全与陪伴守护系统”，通过家庭边缘硬件、云端服务、视觉算法和用户端 App，持续感知老人居家状态，在异常发生时把可理解、可确认、可追踪的提醒推送给家属。

## 2. 产品愿景

`回家` 的核心不是“让子女看监控”，而是“把老人居家状态转化成可行动的牵挂提醒”。

产品要解决三个问题：

- 子女无法长期知道老人独自在家是否安全。
- 普通摄像头只能直播或录像，不能把视频转化为风险事件。
- 老人安全提醒需要在隐私、误报、实时性和可解释之间取得平衡。
- 异地子女和老人之间缺少低负担、可持续的日常牵挂触点，产品不能只在异常时出现。

商业化后的产品应该做到：

- 老人家里放置一个边缘守护盒或预装服务的终端。
- 终端连接本地摄像头、毫米波雷达或其他传感器。
- 视频和传感器数据优先在本地处理，减少隐私暴露。
- 云端只接收事件、结构化结果、必要截图或短视频片段。
- 子女端 App 接收提醒、查看证据、处理事件、配置规则，并每天看到可理解的亲情关怀卡片。
- 平台端支持设备运营、模型版本、告警质量和售后诊断。

## 3. 商业化形态

### 3.1 目标客户

第一阶段目标客户：

- 独居老人家庭
- 子女不在同城的家庭
- 有跌倒、走失、慢病风险的老人家庭

第二阶段可扩展客户：

- 社区养老服务站
- 居家养老上门服务机构
- 养老院和日间照料中心
- 保险、健康管理和银发服务渠道

### 3.2 产品包

商业化产品可以拆成三类：

- 家庭版：一台边缘盒 + 1 到 3 路摄像头 + 子女端 App。
- 社区版：多老人家庭接入 + 社区坐席管理台 + 家属通知。
- 机构版：多房间多摄像头 + 分级告警 + 值班工作台 + 运营报表。

### 3.3 收费模式

可选商业模式：

- 硬件一次性售卖 + 云服务订阅。
- 设备租赁 + 月服务费。
- 社区或机构按老人数量、设备数量或告警服务等级收费。
- 高级算法能力、短信电话通知、人工坐席作为增值服务。

### 3.4 产品边界

必须明确：

- 本产品是居家安全辅助提醒工具，不是医疗诊断设备。
- 跌倒、异常活动、无人活动等结果均为算法候选，需要人工确认。
- 产品价值在于尽早提醒和降低遗漏，不承诺 100% 识别所有事故。
- 隐私设计优先于直播能力，默认不鼓励长期打开视频监控。

### 3.5 商业版本定义

为了避免把阶段验证误认为正式产品，版本定义如下：

| 版本 | 定位 | 目标 | 典型形态 |
| --- | --- | --- | --- |
| V0 盒子链路验证版 | 技术可行性验证 | 跑通 RTSP 摄像头、检测、事件、Web 展示、临时通知和本地盒子安装链路 | Mac 开发对照 + 树莓派 `edge-agent` + Web / Admin 原型 |
| V1 家庭试点版 | 小规模真实家庭试点 | 验证边缘盒稳定性、RTSP 兼容、告警质量、家属使用流程 | 边缘盒 + 云端最小平台 + H5/App 原型 |
| V2 商业家庭版 | 可销售产品 | 支持设备绑定、正式推送、售后诊断、订阅服务 | 量产边缘盒 + 正式 App + 云端平台 |
| V3 社区 / 机构版 | 多设备运营 | 支持多老人、多坐席、告警升级、运营报表 | 设备集群 + 运营后台 + 服务套餐 |

当前开发处于 V0 到 V1 之间。所有本机代码都应该服务于 V1/V2 的架构演进，不能只为了本地验证写死。

### 3.6 当前实施策略

当前阶段采用“树莓派盒子优先验证，Mac 保留开发对照”的策略。

- Mac 继续作为开发、调试和性能对照环境，不能再被描述为正式产品主形态。
- 树莓派是当前阶段的主要验证对象，必须证明它能像一台真实家庭盒子一样独立启动、联网、接摄像头、运行检测、生成事件、展示预览、记录日志和触发报警。
- 当前优先接入局域网 RTSP 摄像头，不再把本机摄像头 `local:0` 作为主线路。
- 盒子侧必须先补齐 `/setup` 配网页、`/admin` 开发管理模式、算法预览、日志诊断和高优先级报警测试。
- 在树莓派盒子侧没有完成自启、稳定拉流、统一实时感知、事件频控和报警测试前，不进入完整云端和正式 App 主开发。

这样做的原因：

- 家庭版产品最终卖的是可通电运行的盒子，不是开发机上的页面。
- Mac 已经证明基础链路可行，下一步关键风险转为树莓派性能、散热、码流、模型负载、服务自启、断线恢复和安装流程。
- 产品核心风险目前集中在“盒子能否独立跑稳、摄像头能否稳定接入、算法结果能否解释、告警能否被家属正确处理”。

### 3.7 标准家庭版使用链路

正式家庭版必须满足下面这条完整链路：

1. 用户拿到守护盒后，先给盒子通电。
2. 如果盒子未联网，用户通过手机连接 `GoHome-XXXX` 热点进入 `/setup`，只完成家庭 Wi-Fi 配网；如果盒子已经联网，直接进入正式 App。
3. 用户在 App 内用手机号登录或创建账号。
4. 用户在 App 内创建或加入家庭空间，并填写老人资料。老人资料未完成前，不能进入完整 App 主功能。
5. 用户在 App 内绑定守护盒。正式产品优先扫盒身二维码；当前树莓派验证阶段可用云端生成的绑定码或设备序列号过渡。
6. 用户在 App 内发起摄像头配置；云端保存目标配置，盒子在老人家局域网内拉取配置并接入 RTSP 摄像头。
7. 老人资料、盒子绑定和至少一路摄像头配置完成后，App 才进入首页、守护、事件、陪伴和我的等主功能。
8. 安装流程完成后，用户手机退出配网页，不再依赖和盒子处于同一局域网。
9. App 通过云端 API 读取设备状态、实时画面、告警、截图、短视频和规则。
10. 用户在任意网络环境下都能继续使用，不要求回到老人家局域网。

这个链路的关键约束是：

- App 不直接访问盒子的局域网 IP。
- App 不直接访问摄像头 RTSP 地址。
- App 的账号、家庭、设备和摄像头配置必须通过云端权限模型绑定；新账号默认不能读取旧家庭数据。
- edge-agent 必须主动连接云端，而不是等 App 反向直连家庭局域网。
- 云端只承接设备状态、事件、截图、短视频、播放会话和规则配置，不直接拉家庭摄像头。
- 未完成配置的新用户只能看到配置向导和退出 / 切换账号动作，不能浏览空首页、空守护、空陪伴或旧家庭数据。
- 盒子绑定和解绑都必须在 App 的设备管理流程完成；盒子本地 `/setup` 只负责 Wi-Fi，不负责家庭归属。

### 3.8 阶段边界说明

为了避免误把当前原型当成正式产品，必须明确区分：

- V0 / 阶段 0：手机或 Web 仍然直接访问本地 `edge-agent`，只能证明“盒子本地视觉链路成立”。
- V1 / 阶段 1-2：盒子具备设备身份、配网、自启、心跳和云端通道，云端可以读取设备状态和事件。
- V1 / 阶段 3：用户正式通过 App 登录云端，在任意网络下使用，不再依赖局域网地址。

因此，正式家庭版的完成标准不是“页面能打开”，而是：

- 盒子可独立部署和恢复。
- 摄像头能在老人家局域网内稳定接入。
- 设备能主动向云端保持心跳并上报事件。
- 用户在异地、任意网络下也能通过 App 查看和处理。

## 4. 用户角色

### 4.1 老人

老人不是主要操作用户，但必须被保护。

关注点：

- 不被复杂设备打扰。
- 不被持续直播侵犯隐私。
- 异常时能尽快被家人知道。
- 设备断网或断电时不要悄悄失效。

### 4.2 子女 / 家属

子女是主要 App 用户。

核心需求：

- 知道家里现在是否平稳。
- 收到异常提醒并快速判断严重程度。
- 查看必要截图或短视频证据。
- 一键确认、误报、联系老人或通知其他家属。
- 调整提醒规则，不被无效告警打扰。
- 每天低负担地知道老人今天是否平稳、有没有值得关心的事。
- 获得合适的问候、提醒和内容建议，但不被无意义内容打扰。

### 4.3 安装和运维人员

核心需求：

- 快速完成设备绑定、摄像头接入和网络配置。
- 看懂摄像头是否在线、算法是否运行、通知是否可达。
- 远程诊断设备状态、日志和版本。
- 安全地升级模型和服务。

### 4.4 平台运营人员

核心需求：

- 查看设备在线率、告警量、误报率。
- 管理用户、设备、套餐、模型版本。
- 处理异常工单和售后问题。
- 监控服务质量和推送送达率。

## 5. 产品级系统架构

商业化系统由五层组成：

```text
用户端 App / Web
-> 云端业务平台
-> 设备连接与 API 网关
-> 家庭边缘硬件 edge-agent
-> 摄像头 / 传感器 / 本地视觉模型
```

### 5.1 用户端

包含：

- iOS App
- Android App
- Web H5 或 WebView 页面
- 家属协作入口

职责：

- 登录和家庭空间管理。
- 绑定守护设备。
- 查看设备和摄像头状态。
- 查看实时摘要、告警列表、告警详情。
- 查看每日亲情关怀卡片和生活节律摘要。
- 接收推送通知。
- 配置看护规则。
- 配置亲情关怀偏好，如提醒频率、老人兴趣、内容来源和生图开关。
- 邀请其他家属。
- 处理告警和反馈误报。

补充要求：

- 配网页和正式使用页必须分开。
- 配网页只负责让盒子连上家庭网络，不承载摄像头、算法、事件、日志和日常使用功能。
- 正式用户端只负责登录、查看状态、处理事件和配置规则，不暴露 RTSP、端口或局域网地址。

### 5.2 云端业务平台

包含：

- 用户与家庭空间服务
- 设备注册和绑定服务
- 摄像头配置服务
- 规则配置服务
- 事件服务
- 媒体服务
- 通知服务
- 亲情关怀与消息生成服务
- 内容推荐与来源管理服务
- 模型编排服务
- 家属协作服务
- 运营管理后台
- API 管理和鉴权

职责：

- 不直接拉取老人家摄像头 RTSP。
- 维护用户、设备、规则、事件和通知状态。
- 接收 edge-agent 上传的结构化结果。
- 向 App 提供统一 API。
- 负责推送、短信、电话等外部通知。
- 基于结构化事件、生活节律、日历、天气、家庭联系记录和用户偏好生成亲情关怀卡片。
- 通过可配置模型 API 生成卡片文案、问候建议和非证据型配图。
- 管理外部内容来源、推荐理由、用户偏好和内容安全策略。
- 管理模型版本、设备版本和远程配置。

### 5.3 边缘硬件端

第一版用当前 M4 / 24GB Mac 模拟商业化边缘盒，商业化后再迁移到独立边缘硬件。

职责：

- 开机自启。
- 首次通电后支持安装模式和配网流程。
- 主动连接云端。
- 连接局域网摄像头。
- 本地拉流、抽帧、缓存。
- 执行基础视觉检测和规则判断。
- 上传事件、截图、短视频和健康心跳。
- 接收云端配置和模型更新。
- 网络异常时本地缓存，恢复后补传。

### 5.3.1 盒子安装模式与本地管理台

商业化盒子不能依赖开发者用 Pi Imager 预填 Wi-Fi。Pi Imager 只允许作为开发和内部验证方式，正式安装必须有用户可操作的安装模式。

盒子必须先有“初始化状态机”，再谈配网和管理。初始化不是联网成功之后才发生，而是在盒子首次启动时先完成本地身份、管理账号和运行目录准备，然后根据网络状态决定是否进入配网。

初始化状态：

| 状态 | 触发条件 | 盒子行为 | 用户可见入口 |
| --- | --- | --- | --- |
| 未初始化 | 没有本地设备身份、管理员密码或初始化标记 | 生成设备 ID、本地密钥、初始管理员凭证、hostname / mDNS 名称、数据目录和日志目录 | 首次启动过程，必要时进入 `/setup` |
| 已初始化未联网 | 已有设备身份，但没有可用家庭 Wi-Fi | 开启 `GoHome-XXXX` 热点，启动 `/setup` 配网页 | 手机连接热点后访问 `/setup` |
| 已联网可管理 | 已有设备身份且已连接家庭网络 | 正常启动 `edge-agent`，关闭或不启用热点，开放 `/admin` 管理入口 | `http://gohome.local/admin` 或局域网 IP |
| 恢复出厂待初始化 | 管理端触发恢复出厂，或硬件长按按钮触发恢复标记 | 清除 Wi-Fi、管理员密码、摄像头配置、算法配置和本地缓存，保留硬件序列号；重启后回到未初始化 | `/setup` |

树莓派本体按钮不能直接等同于产品初始化按钮。开发阶段可以只通过 SSH 或 `/admin` 触发恢复出厂；产品化阶段需要增加以下至少一种方式：

- `/admin` 内的“恢复出厂设置”操作。
- 盒子外壳上的独立长按按钮，接 GPIO，长按 8 到 10 秒后写入恢复标记并重启。
- App / 运维后台下发恢复指令，但必须有权限校验和二次确认。

开发阶段初始化口径：

- 当前树莓派已通过 Pi Imager 配好 Wi-Fi 时，仍然需要执行一次盒子初始化，用来生成设备身份、管理员初始密码、hostname / mDNS 名称和本地数据目录。
- 如果已经能 SSH 进入盒子，可以先走“已联网可管理”路径测试 `/admin`，不必强行断网进入热点。
- Wi-Fi 热点配网作为后续必须补齐的产品化能力，测试时要单独验证“清掉 Wi-Fi 后是否能自动进入热点配网”。

盒子首次通电或网络失效时，应支持至少一种配网方式：

- Wi-Fi 热点配网：盒子发出 `GoHome-XXXX` 热点，手机连接后进入本地 `/setup`。
- BLE 配网：App 扫描盒子蓝牙广播，通过 BLE 写入家庭 Wi-Fi 和绑定凭证。
- 有线兜底：接入网线后进入本地安装页或云端绑定流程。

第一版优先级：

1. 树莓派和 Mac 验证阶段先允许 Pi Imager 或手工 `.env` 配置。
2. 盒子试点前必须补 Wi-Fi 热点配网。
3. BLE 配网作为产品化增强能力，不阻塞第一轮树莓派验证。

设备绑定和认领规则：

- 每台正式盒子出厂前必须生成稳定 `device_id`、设备序列号、设备密钥和一次性认领凭证。
- 盒身、包装盒和说明卡上的二维码只承载设备序列号和认领凭证，例如 `gohome://bind?sn=GH-XXXXXX&claim=...`，不承载局域网 IP、RTSP 地址或摄像头密码。
- App 扫码或手动输入绑定码后，只把 `sn + claim` 提交给云端；云端确认设备存在、未被绑定、凭证有效，再把设备绑定到当前家庭。
- 绑定成功后，认领凭证失效；需要售后重绑或转让时，必须通过解绑、恢复出厂或运营后台重新签发凭证。
- App 解绑只解除云端家庭归属，不清除 Wi-Fi，也不让盒子停止连接云端。解绑后盒子保持在线并恢复为 `claimable`。
- 解绑必须由家庭所有者二次确认，并移除该盒子在旧家庭下的摄像头接入配置；旧家庭历史事件和关怀记录可保留，但不得继续收到该盒子的画面、配置和新事件。
- 盒子上报的 `family_id` 不能覆盖云端归属。设备归属只允许由 App 认领、App 解绑或平台运维操作改变。
- 当前树莓派验证阶段没有实体二维码贴纸时，允许由云端或 `/admin` 显示临时绑定码，用于替代扫码完成同一条认领链路。
- 局域网发现和 BLE 只作为辅助发现或配网能力，不能作为设备归属的唯一依据。
- 云端可以看到未绑定设备在线，但普通 App 用户在完成认领前不能读取该设备的摄像头、事件、截图、规则和历史数据。

盒子本地页面分两类：

- `/setup`：手机优先的配网页，只用于热点模式下选择家庭 Wi-Fi、填写密码、保存连接、提示手机回到家庭网络或 App。
- `/admin`：开发管理模式，供安装人员、研发、售后和高级管理员在盒子入网后登录，管理摄像头、算法预览、报警测试、日志诊断和高级配置。

`/setup` 不允许承载以下内容：

- 摄像头列表、RTSP 明文配置和实时画面。
- 算法开关、算法预览、检测框、阈值和模型内部输出。
- 事件列表、日志列表、规则配置和报警测试。
- 跳转到普通用户日常页面的复杂入口。

配网成功后，`/setup` 只显示下一步入口：

- 普通用户：请让手机回到同一家庭 Wi-Fi，并打开 `回家` App 继续绑定。
- 管理人员：不从普通配网页进入；通过安装文档、调试工具或局域网地址访问 `http://gohome.local/admin` 开发管理模式。

管理地址获取方式必须有三层兜底：

1. mDNS：优先使用 `http://gohome.local/admin`。
2. App / 调试工具：通过设备发现或绑定流程回传盒子局域网 IP。
3. 安装人员兜底：从路由器 DHCP 列表或 SSH 命令查看 IP。

管理登录要求：

- 初始用户名为 `admin`。
- 开发阶段初始密码统一为 `123456`。
- 正式产品初始密码来自设备贴纸、装箱卡或安装人员初始化流程，不得继续使用 `123456` 作为默认密码。
- 正式产品首次登录必须要求修改密码；当前开发演示盒子允许 `admin / 123456` 直接登录，避免每次白纸重测都卡在改密流程。
- 管理模式入口和普通 App 用户入口必须分离。

本地管理台必须和家属端 App 分工明确：

- 本地管理台可以暴露 RTSP、模型阈值、实时算法输出和诊断日志。
- 家属端 App 不暴露 RTSP 密码、底层模型参数和大段运行日志。
- App 只展示用户可理解的规则开关、事件解释、证据和应急动作。
- `/admin` 按桌面 Web 后台设计，不再套手机端页面壳；只有 `/setup` 需要按手机安全区和热点配网场景设计。
- `/admin/cameras` 只承接本地安装调试：扫描局域网设备、填写 IP / 端口 / 用户名 / 密码，选择频道和主/副码流，测试抓帧、保存启用、启停和删除。默认频道 `1`、码流 `2 副码流`，生成路径 `/1/2`。
- 摄像头配置的长期产品源头是云端 / App。V0 阶段 `/admin` 写本地配置用于跑通盒子；V1 后 App / 云端配置生成版本号，edge-agent 拉取或接收配置后应用，本地管理台只显示同步状态、允许安装人员临时覆盖并把覆盖记录回传云端。
- 摄像头测试必须从干净状态开始：默认不自动创建演示摄像头，测试前清空上一张测试截图和指标，失败时不能继续展示旧成功结果。

### 5.4 视觉算法层

包含：

- 视频接入
- 抽帧
- 图像质量检测
- YOLO 人形检测
- 多帧跟踪
- 姿态或动作模型
- 动作识别候选，如用餐、久坐、夜间活动
- 火灾/烟雾/明火视觉候选
- 规则引擎
- 事件融合

算法层不能只等同于“跑一个 YOLO”。YOLO 是目标检测基础能力，产品级异常判断需要把模型结果、时间窗口、空间区域、摄像头状态和业务规则组合起来。

### 5.4.1 统一实时感知与演示要求

盒子本地管理台必须提供一个“统一实时感知”页面，用于安装验收、现场调试和产品演示。页面不再要求研发或安装人员选择单个算法，也不使用独立示意动效卡片；内部算法并行输出到同一个真实摄像头画面。

基本交互：

```text
选择摄像头
-> 自动运行统一感知分析
-> 同一画面叠加人物框、人物编号、姿态骨架、姿势和场景家具
-> 同一画面显示跌倒、火灾和摄像头异常状态
-> 旁侧汇总当前目标、运行指标、最近安全记录和生活观察
-> 支持抓取当前真实检测帧用于研发复核
```

统一感知必须使用当前摄像头实时帧或最近实时分析结果，不能用静态示意图、公开数据集或历史事件图片冒充当前画面。YOLO、RTMPose、场景识别、火灾、时序和云端复核仍保持独立代码边界，但不得在产品页面拆成多个预览模块。

首版统一感知能力：

- 图像质量：亮度、对比度、清晰度、黑屏、遮挡。
- 人形检测：人数、人框、模型置信度；低置信模型候选和肤色、Haar、轮廓等启发式复核必须单独标为“候选分”，不能包装成模型置信度。
- 长时间无人：当前无人时长、阈值和候选事件状态。
- 长时间静止 / 久坐：基于运动变化、人体位置和时间窗的候选判断。
- 疑似跌倒：基于人框比例、位置、姿态或后续姿态模型的候选判断。
- 用餐行为候选：饭点时间、厨房/餐桌区域、人形存在和低移动组合判断。
- 夜间异常活动：夜间时间窗、活动检测和家庭作息基线。
- 火灾候选：烟雾、明火、异常亮度变化或火焰颜色区域的视觉候选。
- 摄像头异常：离线、黑屏、花屏、拉流延迟和最后成功取帧时间。

边界要求：

- “吃饭”第一版只定义为“用餐行为候选”，不能承诺精准识别真实进食动作。
- 火灾视觉结果只定义为“火灾候选”或“烟雾/明火候选”，不能替代烟感、温感或消防报警器。
- 高风险事件必须显示证据、规则命中原因和误报反馈入口。
- 人数只统计模型直接确认或被可信姿态骨架复核通过的人体；家具纹理、肤色区域和 Haar 候选不能计入人数。
- 管理台展示“置信度”时必须保留模型原始分数和模型来源，不允许为了让结果看起来更可靠而人为抬高分数。
- 普通用户只需要选择摄像头安装房间，不要求在 App 里手动画床、沙发或地面区域。盒子应复用场景模型自动识别固定家具，连续多帧稳定后形成每路摄像头的自动场景图。
- 自动场景区域只用于规则解释和误报抑制；管理台可以展示识别框供研发复核，但不能把画区域变成普通用户的必做配置步骤。

### 5.4.2 姿态、动作与模型复用原则

姿态和动作识别不能从零手写。当前阶段优先复用成熟开源或官方模型，先把工程链路跑稳，再根据真实家庭样本做训练和微调。

选型优先级：

1. RTMLib + RTMPose：当前 Pi5 主线。它复用 OpenMMLab 姿态模型，走 ONNX Runtime，适配 Python 3.13 和 Linux arm64，优先用于“框 + 骨架 + 坐/躺/倒地候选”的演示闭环。
2. MoveNet：作为备用轻量姿态路线，用于 RTMPose 在 Pi5 CPU 上帧率不足时降级验证。
3. Hailo AI HAT+ / Hailo Pose：作为产品化边缘推理路线，用 NPU 承担实时姿态和目标检测负载。
4. YOLO11 Pose / YOLO Pose：仅作为低成本备用。它和当前 YOLO 检测链路一致，但骨架稳定性和展示效果不是当前首选。
5. 自训练模型：放到真实样本闭环之后，不能在没有家庭场景样本、误报反馈和标注规范之前直接进入大规模训练。

当前不直接训练的原因：

- 公开数据集可以提供基础人体、姿态和动作能力，但无法覆盖老人家庭里的半身、坐姿、遮挡、背影、低光、沙发/床/轮椅、摄像头俯视角等关键场景。
- 产品成败取决于误报和漏报，不是一次演示命中；必须先有截图、短视频、规则结果和误报反馈，才能形成有效样本库。
- Pi5 性能有限，算法链路必须先解决单路取帧、分析频率、模型加载和页面预览延迟，再叠加更重模型。

### 5.4.3 云端多模态识别编排

商业化后的云端不能只做“存事件”，还需要承接多模态结果编排。

这里的多模态不是指云端直接拉原始摄像头，而是组合这些输入：

- 边缘端上传的结构化视觉结果
- 截图或短视频片段
- 房间、时间段、家庭规则等上下文
- 历史事件和生活节奏基线
- 用户处理记录与误报反馈

云端多模态编排层负责：

- 把边缘端的 `DetectionResult / RuleEvaluation / EventCandidate` 进一步归并成用户可理解的产品事件。
- 生成“是否异常”“是否值得提醒”“提醒给谁”“是否需要升级”的判断。
- 在非高危场景下生成更陪伴型的解释文案，而不是只返回硬告警字段。
- 为后续大模型或多模态模型预留编排层，但第一版不强依赖重模型在线推理。

边界要求：

- 云端默认不直接保存长期原始视频。
- 云端多模态层优先消费结构化结果和必要证据媒体。
- 任何多模态生成结果都必须可追溯到来源事件和规则，不允许黑盒文案直接替代产品证据链。

### 5.4.3 应急报警与处置动作

跌倒、火灾、长时间无响应、摄像头关键失效属于高优先级安全事件，不能只进入普通事件列表。

应急报警链路必须支持：

```text
边缘盒本地生成候选
-> 保存截图或短视频证据
-> 本地记录和本地可见报警状态
-> 云端事件入库
-> App 推送
-> 短信/电话/机器人通知或其它升级通道
-> 超时未处理时通知第二联系人
-> 用户确认、误报或关闭事件
```

事件详情必须给出可执行应急动作：

- 疑似跌倒：查看证据、实时查看、打电话、通知其他家属、联系邻居/物业、拨打急救电话、标记误报。
- 火灾候选：查看证据、联系老人、通知邻居/物业、拨打火警电话、标记误报。
- 长时间无响应：打电话、查看实时画面、通知家庭成员、请求邻居确认。
- 摄像头离线或黑屏：检查电源和网络、联系家里人、标记已处理。

盒子侧可选本地应急能力：

- 蜂鸣器或语音提示。
- 状态灯。
- 断网时本地持续报警和事件缓存。
- 恢复联网后补传应急事件。

### 5.5.1 日志与诊断链路

正式产品必须具备从边缘端到云端再到运营侧的日志链路。

日志来源包括：

- `edge-agent` 运行日志
- 摄像头接入与拉流日志
- 检测与规则执行摘要
- 设备注册、绑定、心跳和同步日志
- 推送送达、失败和回执日志
- 用户查看、确认、误报和处理动作审计日志

日志链路的产品目标：

- 安装人员能快速定位“为什么看不到画面”。
- 开发和运维能定位“为什么没出事件”或“为什么推送没送达”。
- 运营后台能看设备在线率、事件质量和推送送达率。
- 审计侧能追踪谁在什么时间查看、处理了什么事件。

隐私要求：

- 日志不能泄露 RTSP 明文密码、用户 token、手机号全量信息。
- 日志导出和报表必须走脱敏规则。

### 5.6 系统职责矩阵

| 子系统 | 拥有的数据 | 执行的逻辑 | 不应该做的事 |
| --- | --- | --- | --- |
| 用户端 App / H5 | 用户可见状态、表单输入、会话态 | 展示、交互、告警处理动作 | 不直接连 RTSP，不跑视觉模型 |
| 云端业务平台 | 用户、家庭、设备、规则、事件、通知、媒体索引 | 鉴权、状态机、通知编排、配置下发 | 不直接拉家庭摄像头，不保存无授权原始视频 |
| 亲情关怀服务 | 生活节律摘要、日历、天气、联系记录、兴趣偏好、模型生成结果 | 生成每日关怀卡片、问候建议和内容推荐候选 | 不替代安全告警，不凭空编造老人状态，不绕过内容授权 |
| API 网关 / 设备通道 | token、连接状态、请求日志 | 鉴权、限流、路由、审计、协议适配 | 不承载业务判断，不保存大量媒体 |
| 边缘端 edge-agent | 本地配置、帧缓存、检测结果、待同步队列 | 拉流、抽帧、模型推理、规则执行、断网缓存 | 不管理用户账号，不暴露公网摄像头 |
| 视觉算法层 | 模型、检测结果、置信度、轨迹 | 图像质量、人形、姿态、时序分析 | 不直接决定通知家属，不替代业务规则 |
| 运营后台 | 设备质量、告警质量、用户反馈、工单 | 售后诊断、模型灰度、数据报表 | 不面向老人日常使用，不绕过权限查看隐私内容 |

### 5.7 端到端数据流

商业化数据流必须这样拆：

```text
摄像头原始视频
-> 边缘端帧缓存
-> 视觉模型 DetectionResult
-> 规则引擎 RuleEvaluation
-> 候选事件 EventCandidate
-> 云端事件 Event
-> 通知 Notification
-> 用户处理 UserAction
-> 反馈 Feedback
-> 模型和规则优化
```

关键约束：

- 原始视频默认不离开老人家。
- 上传云端的是事件、结构化检测结果、必要截图或短视频。
- App 展示的是产品化提醒，不展示算法内部对象。
- 运营后台看到的是质量指标和诊断信息，不默认暴露隐私媒体。

亲情关怀数据流必须这样拆：

```text
设备状态 / 生活节律 / 事件摘要 / 日历 / 天气 / 联系记录 / 老人兴趣偏好
-> care-service 生成关怀上下文
-> model-service 可选调用文本模型生成候选文案
-> image-service 可选调用生图模型生成非证据型配图
-> content-service 可选召回合规内容链接
-> message-service 生成 MessageCandidate / CareCard
-> notification-service 按频率和优先级推送
-> App 展示卡片、记录查看和反馈
```

关键约束：

- 亲情关怀不能假装是实时医学判断，也不能替代安全告警。
- 每张卡片必须能说明来源，如“今日设备在线”“下午有活动”“天气降温”“你设置的生日提醒”。
- 由模型生成的文字必须有规则兜底，模型不可用时仍能生成基础卡片。
- 生图只用于温暖表达、节日氛围、问候卡片等非证据内容；不能替代真实截图、告警证据或老人真实影像。
- 外部视频、公众号文章等内容推荐必须基于用户授权、合规来源、可追溯链接和频率控制；第一版不做未经授权的抓取和全文搬运。

## 6. 模块职责

### 6.1 用户端 App

P0 商业功能：

- 手机号登录 / 注册
- 创建家庭空间
- 加入家庭空间
- 绑定守护盒
- 设备在线状态
- 摄像头列表
- 最新状态摘要
- 实时画面查看
- 告警列表
- 告警详情
- 截图或短视频查看
- 告警处理：已确认、误报、稍后提醒
- 看护规则开关
- 推送通知
- 回家消息与陪伴消息卡片
- 每日亲情关怀卡片
- 亲情关怀偏好设置：频率、兴趣标签、内容开关、生图开关

当前 P0 不把微信登录、Apple 登录等第三方登录作为已实现能力；只有真实接入后才能出现在用户端入口中。

P1 功能：

- 邀请家属
- 告警升级策略
- 联系老人快捷入口
- 家属处理记录
- 设备安装向导
- 模型生成卡片配图和节日问候图
- 外部内容推荐：合规视频、公众号文章、图文链接

### 6.2 产品 Web 前端

当前 Web 原型承担两个职责：

- 阶段 0：作为 App 前的产品验证界面。
- 阶段 1：作为 H5 / WebView / 管理辅助页面复用。

商业化后需要拆分：

- 用户端 H5：给家属看状态、告警和规则。
- 安装调试页：给安装人员接入摄像头。
- 运营后台：给平台运营和售后诊断。

补充要求：

- 页面端和 App 端都必须通过同一套后端能力完成核心流程，不能只做网页能看、App 不能看。
- 实时画面能力必须同时支持页面端和 App 端，不允许把“实时查看”只留在后台调试页。
- 页面端和 App 端共享同一套用户语义：房间、实时状态、事件、规则、证据，而不是共享边缘端内部调试字段。

### 6.3 云端后端

核心服务：

- `identity-service`：用户、家庭、角色和权限。
- `device-service`：设备注册、绑定、心跳、版本。
- `camera-service`：摄像头配置、状态和接入能力。
- `video-service`：实时画面代理、播放鉴权、会话控制、码流选择。
- `rule-service`：规则模板、家庭规则、下发版本。
- `event-service`：事件入库、查询、状态流转。
- `media-service`：截图、短视频、留存和访问授权。
- `notification-service`：App 推送、短信、电话、机器人通知。
- `model-service`：模型版本、能力开关、灰度策略。
- `care-service`：每日关怀上下文、生活节律摘要、问候建议和卡片编排。
- `content-service`：内容来源、兴趣偏好、候选召回、推荐理由和内容安全。
- `image-service`：非证据型卡片配图、生图任务和平台模型能力接入。
- `ops-service`：运营报表、设备诊断、审计日志。

补充服务职责：

- `message-service`：把事件、节奏、回忆和陪伴语义组合成“回家消息”“提醒消息”“解释消息”。
- `log-service`：接收边缘端运行摘要、推送回执和关键审计日志，供诊断与运营分析使用。

模型编排要求：

- `model-service` 不能直接修改事件、设备和规则状态，只能生成候选结果。
- 亲情关怀的长期体验以模型 API 增强为主，但第一版每日卡片必须先由结构化事实和模板规则生成，避免模型未配置时产品不可用。
- 文案模型第一版用于“每日摘要、问候建议、卡片标题和解释文案”，必须带模板兜底。
- 生图模型第一版作为平台侧模型能力接入，例如 `wan2.7` 或等价图像生成模型；没有平台密钥或生成失败时使用默认卡片样式。
- 模型调用必须记录 `provider / model / prompt_version / input_hash / output_status`，便于成本控制、复盘和替换。
- 高危安全事件不依赖大模型生成才能推送；模型只负责增强解释，不负责决定是否报警。
- 模型 API key、Base URL、模型名和系统提示词由平台方配置，不属于家属用户可配置项。
- 本地开发优先通过服务器环境变量配置模型能力；云端必须接 Secret Manager / KMS，数据库不保存明文 API key。
- 运营后台只展示平台内部模型能力状态和 env 配置指引，不提供给普通用户填写 key、Base URL、模型名或 Prompt。

内容推荐要求：

- 第一版只推荐链接和摘要，不复制外部平台全文或搬运视频。
- 来源优先级为：用户手动订阅 / 白名单来源 / 合规公开 API 或 RSS / 人工运营配置。
- 公众号文章、短视频和自媒体内容属于 P1.5/P2，必须先解决来源授权、内容安全、老人兴趣偏好和推送频控；自动搜索与跨平台召回不能进入当前本地闭环优先级。
- 推荐理由必须可解释，例如“你为妈妈设置了养生和越剧兴趣”“今天杭州降温，适合提醒添衣”。

`message-service` 的第二批输入域必须预留：

- `ElderProfile`：老人资料档案，如称呼、关系、城市、手机号、家里电话、生日、喜好、作息和敏感健康备注。
- `CalendarEvent`：家庭日历和重要日期，如生日、节日、体检、回家计划。
- `WeatherSignal`：天气和外部环境信号，如雷暴雨、降温、高温和空气质量。
- `ContactRecord`：联系记录，如最近一次通话、消息或手动联系记录。
- `VisitRecord`：回家或陪伴记录，如最近一次回家、计划中的回家安排。

这些对象的定位不是替代 `Event`，而是作为 `MessageCandidate` 的补充输入，让云端能够把“安全事件”进一步转成“场景化图文消息卡片”。

### 6.4 edge-agent

商业化 edge-agent 不是单个脚本，而是边缘设备运行时。

模块：

- `setup-agent`：首次通电、Wi-Fi 热点配网、BLE 配网预留和配网成功后的回到 App / 管理端引导。
- `device-agent`：设备身份、心跳、云端连接、远程配置。
- `camera-agent`：ONVIF 扫描、RTSP 拉流、断线恢复。
- `stream-agent`：FFmpeg / OpenCV 拉流、帧缓存、短视频缓存。
- `live-agent`：实时画面转发、子码流选择、实时会话限流。
- `vision-agent`：YOLO、图像质量、姿态、跟踪。
- `preview-agent`：算法预览、单算法实时效果、检测框和规则解释。
- `rule-agent`：时间窗、区域、阈值、家庭规则。
- `event-agent`：事件生成、去重、节流、补传。
- `alert-agent`：高优先级报警、升级策略、测试报警和本地应急状态。
- `media-agent`：截图、短视频、脱敏、水印和本地留存。
- `diagnostic-agent`：运行日志、摄像头拉流错误、模型错误、诊断包导出。
- `update-agent`：服务升级、模型升级、回滚。
- `watchdog`：进程保活、异常重启、磁盘清理。

### 6.5 硬件端

商用品类：

- 当前：M4 / 24GB Mac，作为阶段 0 本地算力服务。
- 早期试点：Mac mini / N100 小主机 / 工控机。
- 低功耗验证：Raspberry Pi 5，优先验证部署、功耗、散热、网络和 24 小时稳定性。
- AI 加速验证：Raspberry Pi AI HAT+ 或同类 NPU 加速模块，在需要树莓派本地高频模型推理时再评估。
- 产品化：无风扇低功耗边缘盒。
- 可选：带 NPU 的国产或 ARM 边缘设备。

硬件要求：

- 长期 24 小时运行。
- 支持有线网络，Wi-Fi 作为备选。
- 支持首次通电后的 Wi-Fi 热点配网，后续预留 BLE 配网。
- 支持至少 1 到 4 路 1080p 摄像头抽帧检测。
- 本地存储事件截图和短视频缓存。
- 断电恢复后自动启动。
- 支持远程升级和日志采集。
- 具备设备唯一身份和安全密钥。
- 可选支持蜂鸣器、状态灯或语音提示，用于高优先级本地应急提醒。

## 7. 视觉模型与算法方案

### 7.1 视觉能力分层

视觉能力分为四层：

| 层级 | 能力 | 技术方案 | 产品用途 |
| --- | --- | --- | --- |
| L0 图像质量 | 黑屏、遮挡、模糊、过暗 | OpenCV 亮度、对比度、清晰度 | 保障摄像头可用 |
| L1 目标检测 | 人、宠物、轮椅、床、沙发 | YOLOv8/YOLOv10/YOLO11 系列 | 判断画面中是否有人 |
| L2 姿态骨架 | 头肩髋膝踝等关键点、坐姿、躺姿、半身 | RTMLib + RTMPose 主线，MoveNet / Hailo 备用 | 让演示从“只有框”升级到“框 + 骨架 + 姿态解释” |
| L3 时序理解 | 停留、离开、长时间无人、活动量下降 | 多帧跟踪 + 状态机 | 降低单帧误判 |
| L4 行为风险 | 疑似跌倒、异常卧地、夜间活动 | 姿态模型 + 目标检测 + 时序规则 | 生成高优先级告警 |

### 7.1.1 算法工程组织要求

为了避免算法逻辑越写越臃肿，视觉能力必须按“单算法单文件”组织：

- 图像质量检测一个文件。
- 人形检测一个文件。
- 姿态检测一个文件。
- 跌倒判断一个文件。
- 区域停留一个文件。
- 夜间活动一个文件。
- 活动量或生活节奏分析一个文件。

工程约束：

- 每种算法负责输出自己的原始结果，不直接改写用户事件。
- 规则引擎负责把多个算法结果组合成业务判断。
- 后台配置页必须能看到每个算法模块是否启用、版本号、关键阈值和最近输出摘要。
- 姿态检测类算法不能只展示框框，还必须能展示关键点、姿态结论、时序状态或风险解释中的至少一种。

### 7.2 当前模型路线

阶段 0 已经使用 YOLO 人形检测和人体存在增强跑通“检测框 + 人数 + 候选标签”的基础链路。下一步不是继续堆单帧 YOLO，而是先做两件事：

1. 把视频输入改成单路帧源，避免预览和算法各自打开 RTSP 导致卡顿。
2. 引入轻量姿态 POC，让算法预览能显示骨架关键点、坐/躺/倒地候选和时序解释。

YOLO 检测继续保留，原因是：

- 模型小，适合当前 Mac 本地算力验证和未来低功耗硬件验证。
- 能快速得到人形检测框、置信度和人数。
- 接入成本低，适合跑通链路。

姿态模型选择：

- RTMLib + RTMPose：当前主线。通过 ONNX Runtime 在 Pi5 Python 3.13 环境运行，优先保证骨架展示质量和可部署性。
- MoveNet：备用轻量路线，用于 RTMPose 帧率或温度不达标时降级。
- Hailo Pose：产品化加速路线，适配 AI HAT+ 后用于实时姿态和更低 CPU 占用。
- YOLO11 Pose：低优先级备用，只在 RTMPose / MoveNet 不可用时评估。
- MediaPipe：当前 Pi5 Python 3.13 环境依赖不可用，不作为主线。
- OpenVINO / TFLite / NCNN：后续用于量化和加速。

产品化阶段继续评估：

- `YOLOv8n`：速度快，适合低算力边缘盒，精度较低。
- `YOLOv8s` / `YOLOv8m`：精度更高，硬件成本上升。
- RTMPose / MoveNet：用于人体关键点、坐卧、长时间卧地和跌倒候选。
- YOLO Pose：备用关键点能力，不作为当前展示主线。
- 模型量化：ONNX / NCNN / OpenVINO / CoreML / TensorRT，视硬件而定。

### 7.3 YOLO 在本产品里的角色

YOLO 只负责回答：

- 画面里是否有人。
- 人在哪里。
- 有几个人。
- 检测框大小、位置、置信度是多少。

YOLO 不直接回答：

- 老人是否安全。
- 是否一定跌倒。
- 是否需要通知家属。

产品级判断必须由规则引擎和时序逻辑完成：

```text
YOLO 检测结果
+ 最近 N 秒轨迹
+ 摄像头所在房间
+ 家庭规则
+ 时间段
+ 历史活动基线
+ 事件去重和置信度
= 产品级提醒
```

### 7.4 核心检测算法

P0 到 P1 的检测逻辑：

- 摄像头离线：RTSP 拉流失败、连续抓帧失败、心跳超时。
- 黑屏 / 遮挡：亮度、对比度、边缘信息和连续帧判断。
- 长时间无画面变化：帧差低于阈值并持续超过规则时间。
- 长时间无人：YOLO 连续 N 帧无 person 且持续超过规则时间。
- 异常卧地候选：人框横纵比、位置、持续时间、地面区域以及床/沙发等正常卧躺区域联合判断。
- 疑似跌倒候选：活动或站坐状态到快速下降、低位卧地持续、无后续起身的时序过程；单帧横躺、睡床或躺沙发不能直接判为跌倒。
- 夜间异常活动：时间段 + 人形出现 + 活动轨迹。
- 区域停留：预设区域 + 停留时长 + 摄像头房间语义。

这些算法在工程实现上必须满足：

- 一种算法一个实现文件。
- 一种算法一套明确输入输出。
- 一种算法一组后台可配置参数。
- 一种算法一份最近结果可查看摘要。
- 跌倒正式事件必须经过连续帧状态机和区域规则确认；公开数据集的单帧命中率只用于回归，不作为开启正式通知的依据。
- 新识别的床/沙发候选必须经过多帧位置稳定，并排除大部分面积被当前人框或骨架覆盖的“人体误识别成家具”结果。
- 正式跌倒至少需要近期站立或坐姿证据、明显向下位移或运动变化、同一目标低位持续，并且不位于稳定床/沙发区域；一次确认过的下降过程可以沿同一人体轨迹继承，不要求每个倒地帧重复下降。

事件输出分层：

- 跌倒、火灾、摄像头离线、黑屏 / 遮挡属于正式事件，进入告警、留证和上传队列。
- 长时间无人、长时间无变化属于生活观察区间，只做开始、持续、恢复的聚合记录，不按固定间隔刷正式事件。
- 生活观察后续可同步到云端老人日志，用于 App 消息、趋势和运营分析，但默认不触发高危报警。

### 7.5 误报治理

商业化能否成立，关键不是“模型能不能识别一次”，而是长期误报是否可控。

必须建立：

- 事件置信度。
- 事件合并和节流。
- 用户误报反馈。
- 家庭级阈值调节。
- 摄像头位置校准。
- 模型版本灰度。
- 告警复盘数据集。
- 可解释证据：截图、时间线、检测框、规则命中原因。

### 7.6 回家消息、每日关怀与推送链路

`回家` 不是只做异常报警，还要做“值得回家的一眼”“今天家里还好吗”和“适合发起联系的理由”。

第一版亲情关怀消息链路定义如下：

```text
边缘端检测结果
-> 云端事件与节奏聚合
-> care-service 生成每日关怀上下文
-> message-service 生成消息候选和 CareCard
-> 规则判断消息类型和优先级
-> notification-service 下发推送
-> App 打开消息详情或相关页面
```

消息类型至少包括：

- 安全异常消息：如离线、黑屏、疑似跌倒候选、疑似火灾
- 需要确认消息：如长时间低姿态、长时间未活动、长时间无人
- 回家消息：如“最近 12 天没回去，这周建议先回家看看”
- 陪伴消息：如“今天适合打个电话”或“有点想她了”
- 每日关怀卡片：如“今天设备在线，下午有活动，天气降温，适合提醒添衣”
- 内容推荐卡片：如“给妈妈推荐一段她爱看的戏曲视频”，只在用户开启后出现

生成要求：

- 回家消息不能凭空生成，必须基于可解释的生活节奏、事件趋势、回忆触发或家庭规则。
- 推送文案和 App 内消息卡片要统一来源，不允许页面写一套、推送写一套。
- 高危异常优先级永远高于陪伴消息。
- 每日关怀卡片第一版必须不依赖大模型也能生成；模型 API 用于润色、摘要和个性化表达。
- 每日关怀不是把 7 类内容拆成 7 条固定推送。第一版默认生成一张“每日汇总卡”，把家里状态、问候话题、天气、日历、回家间隔和纪念日合并展示。
- 推送规则必须按内容类别拆分保存：家里异常可即时提醒，节日可提前 N 天提醒，纪念日按每年同月同日提醒，回家间隔按阈值提醒。
- 生图只用于 `accompany / gohome / daily / festival` 等非证据卡片，不用于告警证据。
- 内容推荐必须默认关闭，由用户选择兴趣和来源后再开启。
- 内容推荐的内容类型、老人兴趣和内容区域必须统一从“我的 -> 关怀推送”读取；老人资料里的城市/区县只作为默认值，不允许首页、陪伴页或消息页各自维护一套偏好。
- 天气信号必须来自平台侧天气 provider。当前本地实现支持和风天气 `qweather`，城市搜索走 `geoapi.qweather.com`，实时天气走 `devapi.qweather.com`；未配置或失败时必须明确降级，不允许编造温度、降雨或实时预报。
- 热点和内容候选必须来自平台侧内容搜索 provider。当前本地实现支持 Tavily Search，定位为“老人兴趣话题/文章视频候选”，只给关怀卡片和家属端内容上下文使用；Tavily 不能替代天气 API，也不能默认向老人推送外链。搜索任务应按用户开启的内容类型拆分，例如本地热点、健康养生、防诈骗和文娱兴趣，而不是用一条泛搜索硬凑首页内容。
- Web 本地阶段不直接跳微信。正式 iOS App 阶段再接微信跳转或系统分享，未接入前不能把“发消息”伪装成已完成外部消息能力。

优先级：

1. P0：每日关怀卡片、问候建议、联系入口、规则模板兜底。
2. P0.5：文本模型 API 生成更自然的卡片文案，失败时回退模板。
3. P1：平台侧生图模型能力，如 `wan2.7`，用于非证据型卡片配图。
4. P1.5：用户手动订阅或白名单来源的内容推荐链接。
5. P2：自动搜索自媒体视频、公众号文章和跨平台内容召回。

暂不优先做 P2 的原因：

- 外部内容平台接口、版权、转发规则和搜索质量不稳定。
- 老人内容偏好需要先由用户显式配置，否则推荐容易变成噪音。
- 内容安全和频率控制没有做好前，外部内容推送会削弱“家人关怀”的产品感。

## 8. 数据与逻辑分层

### 8.1 分层原则

商业化架构必须把数据和逻辑拆开：

- 原始视频：默认留在边缘端，不上传云端。
- 截图和短视频：只在事件触发时上传，且有留存策略。
- 检测结果：结构化上传，用于 App 展示、统计和误报治理。
- 业务规则：云端配置，边缘端执行。
- 用户状态：云端保存，App 查询。
- 设备运行状态：边缘端上报，云端用于诊断。

额外约束：

- 页面和 App 不承载算法判断代码。
- 页面和 App 不拼接复杂业务状态机。
- 后端不把展示文案、页面结构和视觉算法写成一个大文件。
- 视频服务、算法服务、规则服务、事件服务必须逻辑分离。

### 8.2 数据对象

核心对象：

- `User`：用户账号。
- `Household`：家庭空间。
- `Member`：家庭成员和角色。
- `ElderProfile`：老人基础信息、风险偏好和联系信息。
- `CalendarEvent`：生日、节日、复诊、回家计划等家庭日历事件。
- `Device`：边缘盒设备。
- `Camera`：摄像头配置和状态。
- `RuleSet`：看护规则集合。
- `WeatherSignal`：天气和外部环境信号。
- `ContactRecord`：通话、消息或手动联系记录。
- `VisitRecord`：回家、陪伴和探访记录。
- `CarePreference`：亲情关怀偏好，如卡片频率、兴趣标签、生图开关、内容来源开关。
- `CareCard`：每日亲情关怀卡片，是 `MessageCandidate` 的用户端聚合展示。
- `ContentSource`：内容来源配置，如白名单账号、RSS、人工运营链接或用户收藏来源。
- `ContentRecommendation`：推荐给家属或老人的内容候选，包含链接、来源、推荐理由和安全状态。
- `ModelGenerationJob`：模型生成任务，如文案生成、生图生成、摘要生成。
- `DetectionResult`：单帧或多帧算法结果。
- `Event`：产品级告警事件。
- `MediaAsset`：截图、短视频、缩略图。
- `MessageCandidate`：回家消息、陪伴消息、解释消息候选。
- `Notification`：推送、短信、电话等通知记录。
- `NotificationReceipt`：推送送达、点击、失败等回执。
- `DeviceLog`：边缘端运行摘要和诊断日志。
- `Feedback`：已处理、误报、严重程度反馈。
- `AuditLog`：配置、查看、处理的审计记录。

### 8.3 事件生成链路

事件链路：

```text
Frame
-> DetectionResult
-> RuleEvaluation
-> EventCandidate
-> Event
-> MessageCandidate
-> Notification
-> NotificationReceipt
-> UserAction
-> AuditLog
-> Feedback
```

说明：

- `DetectionResult` 是算法事实。
- `RuleEvaluation` 是规则命中过程。
- `EventCandidate` 是候选事件，可被合并或抑制。
- `Event` 是用户端可见事件。
- `MessageCandidate` 是面向家属端的解释/陪伴/回家消息层，不替代异常事件。
- `MessageCandidate` 可以由事件、节奏、老人资料、天气、日历、联系记录和回家记录共同生成。
- `NotificationReceipt` 是通知结果层，用于送达追踪和失败分析。
- `AuditLog` 是责任追踪层，用于查看、播放、确认、忽略等关键动作留痕。
- `Feedback` 用于后续规则和模型优化。

### 8.3.1 核心数据契约

为了让边缘端、云端、App 和运营后台可以并行演进，以下对象必须从阶段 0 开始按统一语义设计。

`DetectionResult` 最少字段：

- `detection_id`
- `camera_id`
- `captured_at`
- `frame_width`
- `frame_height`
- `detector_backend`
- `model_name`
- `model_version`
- `person_count`
- `objects`
- `quality_flags`
- `raw_confidence_summary`

说明：

- `objects` 至少包含类别、置信度、边界框。

`MessageCandidate` 最少字段：

- `message_id`
- `family_id`
- `device_id`
- `message_type`
- `priority`
- `title`
- `body`
- `source_event_ids`
- `source_media_ids`
- `generated_by`
- `created_at`
- `status`
- `render_payload`

说明：

- `message_type` 至少支持 `alert / explain / accompany / gohome`。
- `generated_by` 用于标记来源于规则、节奏分析还是人工运营。
- `MessageCandidate` 必须引用来源事件、媒体、日历、天气、联系记录、家庭规则或内容来源，不能凭空生成。
- `render_payload` 用于卡片渲染，不替代核心字段。

`MessageCandidate` 的卡片渲染补充字段允许在展示层存在，但不应替代核心对象：

- `subtitle`
- `facts`
- `image_mode`
- `image_url`
- `actions`
- `source_summary`
- `content_recommendations`

这些字段用于把 `MessageCandidate` 渲染成“场景化图文消息卡片”，产品上可见，但系统契约层仍以 `MessageCandidate` 为中心。

`CareCard` 最少字段：

- `card_id`
- `family_id`
- `elder_id`
- `card_date`
- `card_type`
- `title`
- `body`
- `facts`
- `source_message_ids`
- `image_mode`
- `image_url`
- `actions`
- `status`

说明：

- `card_type` 至少支持 `daily / weather / contact / festival / content / reassurance`。
- `facts` 必须来自结构化事实或用户配置，不允许模型自行编造老人行为。
- `image_url` 只能用于非证据配图；真实告警证据仍走 `MediaAsset`。

`ModelGenerationJob` 最少字段：

- `generation_id`
- `family_id`
- `job_type`
- `provider`
- `model`
- `prompt_version`
- `input_hash`
- `status`
- `output`
- `error`
- `created_at`

说明：

- `job_type` 至少支持 `care_text / care_image / content_summary`。

`ContentRecommendation` 最少字段：

- `recommendation_id`
- `family_id`
- `elder_id`
- `source_id`
- `title`
- `url`
- `summary`
- `content_type`
- `reason`
- `safety_status`
- `created_at`

说明：

- `content_type` 至少支持 `video / article / audio / activity`。
- 第一版只保存链接、标题、摘要和推荐理由，不保存外部平台全文或视频文件。

`NotificationReceipt` 最少字段：

- `receipt_id`
- `notification_id`
- `channel`
- `receipt_type`
- `provider_message_id`
- `occurred_at`
- `detail`

说明：

- `receipt_type` 至少支持 `accepted / delivered / clicked / failed`。
- `NotificationReceipt` 与 `Notification` 是一对多关系。

`DeviceLog` 最少字段：

- `log_id`
- `device_id`
- `log_type`

- `message`

- `context`

说明：

- `context` 用于附带拉流错误、检测摘要、同步错误等结构化字段。

`AuditLog` 最少字段：

### 8.3.2 契约稳定性要求

从阶段 1 开始，核心对象不只是“文档概念”，而是正式产品契约：

- 数据库表命名应与核心对象语义保持一致。
- OpenAPI 中的 schema 名称应与核心对象保持一致。
- 错误码必须围绕这些对象和它们的阶段边界设计，不允许页面自己发明业务错误语义。
- 如果对象命名、字段或状态机发生变化，必须先更新 `PRD`，再更新 `Plan` 和实现。

目标是确保：

- 页面看到的对象名
- 接口返回的 schema 名
- 表结构中的业务语义
- 日志、审计和消息链路中的对象关系

始终是同一套产品语言。

`EventCandidate` 最少字段：

- `candidate_id`
- `camera_id`
- `event_type`
- `candidate_level`
- `started_at`
- `ended_at`
- `dedupe_key`
- `source_evaluations`
- `evidence_snapshot_ids`
- `status`

说明：

- `dedupe_key` 用于边缘端和云端去重。
- `status` 至少支持 `new / merged / suppressed / promoted / expired`。
- `EventCandidate` 允许被合并、抑制或晋升为用户可见事件。

`Event` 最少字段：

- `event_id`
- `household_id`
- `device_id`
- `camera_id`
- `event_type`
- `level`
- `status`
- `occurred_at`
- `resolved_at`
- `summary`
- `reason_text`
- `evidence`
- `notification_status`
- `feedback_status`

说明：

- `status` 至少支持 `open / acknowledged / false_positive / ignored / closed`。
- `evidence` 至少包含截图、短视频、检测框、命中规则摘要中的一种或多种。
- `Event` 是唯一面向家属和客服可见的业务事件对象。

对象关系约束：

- 一个 `DetectionResult` 可以对应 0 到多个 `RuleEvaluation`。
- 多个 `RuleEvaluation` 可以汇总成一个 `EventCandidate`。
- 一个 `EventCandidate` 最多晋升为一个用户可见 `Event`。
- 一个 `Event` 可以关联多个 `Notification` 和多个 `Feedback`。

### 8.4 数据适配

前端、App、云端和边缘端不能直接共享内部结构，必须通过适配层：

- 边缘端内部：以帧、检测结果、规则状态为主。
- 云端内部：以家庭、设备、事件、媒体、通知为主。
- 用户端接口：以“家里状态、提醒、可执行动作”为主。
- 运营后台接口：以设备质量、告警质量、模型效果为主。

实时画面也必须通过适配层：

- 边缘端负责拉流和转码控制。
- 视频服务负责鉴权、会话、码流选择和播放地址下发。
- 页面端和 App 端只消费被授权的实时画面地址或会话，不直接持有 RTSP 明文。

## 9. API 管理

API 必须从第一阶段就按产品化方式设计。

### 9.1 API 分类

- 用户端 API：App / H5 使用。
- 设备端 API：edge-agent 与云端通信。
- 管理端 API：运营后台和安装调试使用。
- 内部 API：模型、事件、通知、媒体服务之间调用。

### 9.2 用户端 API

核心领域：

- 登录和身份。
- 家庭空间。
- 设备绑定。
- 摄像头列表。
- 首页摘要。
- 实时画面会话。
- 告警列表。
- 告警详情。
- 告警处理。
- 看护规则。
- 家属协作。
- 媒体访问授权。

### 9.3 设备端 API

核心领域：

- 设备注册。
- 设备鉴权。
- 心跳上报。
- 配置拉取。
- 摄像头状态上报。
- 实时流能力上报。
- 事件上报。
- 媒体上传。
- 日志上报。
- 版本更新。
- 模型更新。

通信策略：

- 设备主动连云端，避免家庭公网 IP 和端口映射。
- 配置和事件使用 HTTPS。
- 实时控制和在线状态可使用 WebSocket 或 MQTT。
- 媒体上传使用预签名 URL 或对象存储凭证。

### 9.4 API 治理

必须支持：

- `/api/v1` 版本命名。
- OpenAPI 文档。
- 设备 token 和用户 JWT 分离。
- 家庭角色权限控制。
- 请求幂等键。
- 事件去重键。
- 限流。
- 审计日志。
- 兼容旧设备版本。

### 9.5 API 领域草案

商业化 API 至少需要这些领域：

| 领域 | 面向对象 | 示例能力 |
| --- | --- | --- |
| Identity API | 用户端 | 登录、刷新 token、退出登录 |
| Household API | 用户端 | 创建家庭、成员列表、角色权限 |
| Device API | 用户端 / 设备端 / 管理端 | 设备绑定、解绑、心跳、版本、在线状态 |
| Camera API | 用户端 / 设备端 / 管理端 | 摄像头配置、状态上报、测试结果 |
| Video API | 用户端 / 设备端 / 管理端 | 实时画面会话、播放地址、码流能力、观看鉴权 |
| Rule API | 用户端 / 设备端 | 规则读取、保存、版本下发、设备确认 |
| Event API | 用户端 / 设备端 | 事件上报、列表、详情、确认、误报 |
| Media API | 用户端 / 设备端 | 上传截图、上传短视频、授权访问 |
| Notification API | 用户端 / 云端内部 | 推送设置、通知记录、送达状态 |
| Model API | 设备端 / 管理端 | 模型版本、灰度策略、能力开关 |
| Ops API | 管理端 | 设备诊断、告警质量、工单、审计 |

API 返回给用户端时，要隐藏技术细节，把字段适配成“房间、时间、提醒原因、建议动作、证据材料”。

### 9.6 API 验收要求

从阶段 1 开始，所有新接口都要满足以下最低要求：

- 有清晰的请求和响应示例。
- 有字段说明、必填约束和默认值说明。
- 有错误码和失败场景说明。
- 有鉴权方式说明，明确是用户 JWT、设备 token 还是内部调用。
- 有幂等规则说明，明确哪些接口必须使用幂等键。
- 有审计要求说明，明确哪些查看和处理动作需要落审计日志。

用户端接口最低约束：

- 不返回 RTSP 密码。
- 不暴露边缘端内部异常堆栈。
- 不直接返回算法内部调试结构，必须先做产品化适配。

设备端接口最低约束：

- 必须支持设备身份校验。
- 必须支持重试和去重。
- 必须允许旧版本设备在灰度期继续工作。
- 必须区分“可重试错误”和“不可重试错误”。

## 10. 前后端职责拆分

### 10.1 用户端前端

前端只负责：

- 展示状态和提醒。
- 发起用户动作。
- 展示截图、短视频和处理建议。
- 展示被授权的实时画面。
- 管理规则配置表单。

前端不负责：

- 直接连接 RTSP。
- 执行视觉算法。
- 判断跌倒是否成立。
- 持久化核心业务状态。
- 在页面里拼接复杂算法和业务判断。

### 10.2 云端后端

云端负责：

- 用户、家庭、设备和权限。
- 规则版本和下发。
- 事件状态机。
- 实时画面会话和播放鉴权。
- 通知编排。
- 媒体访问授权。
- 多端同步。
- 运营后台和审计。

### 10.3 边缘端

边缘端负责：

- 摄像头连接。
- 本地视觉计算。
- 本地规则执行。
- 本地缓存和断点续传。
- 实时画面能力供给。
- 与云端保持设备通道。

## 11. 核心用户流程

### 11.1 设备安装

```text
安装人员通电边缘盒
-> 如未联网，手机连接盒子热点并打开 /setup
-> /setup 只完成家庭 Wi-Fi 配网
-> 手机回到家庭 Wi-Fi 或打开回家 App
-> App 手机号登录或注册
-> App 创建或加入家庭
-> App 填写老人资料
-> App 扫盒身二维码或输入绑定码认领设备
-> edge-agent 上报设备在线
-> App 添加或扫描摄像头配置
-> 盒子从云端拉取配置并在老人家局域网内测试画面
-> App 配置房间和规则
-> 完成安装
```

### 11.2 日常守护

```text
edge-agent 本地抽帧
-> 视觉模型生成检测结果
-> 规则引擎判断是否异常
-> 本地保存截图 / 短视频
-> 上报事件到云端
-> 云端推送 App
-> 家属查看并处理
```

### 11.2.1 实时画面查看

```text
家属在页面或 App 点击查看实时画面
-> 用户端向云端申请实时会话
-> 云端校验用户、家庭、设备和摄像头权限
-> 视频服务向 edge-agent 请求可用码流
-> edge-agent 返回被授权的实时播放能力
-> 页面 / App 播放实时画面
```

约束：

- 页面端和 App 端都必须跑通这个流程。
- 实时画面不能绕过后端权限体系。
- 实时画面能力属于正式产品能力，不是仅限后台调试能力。

### 11.3 告警处理

```text
家属收到推送
-> 打开告警详情
-> 查看截图 / 短视频 / 命中原因
-> 选择已确认 / 误报 / 稍后提醒
-> 可联系老人或通知其他家属
-> 处理结果回写云端
-> 反馈进入规则和模型优化闭环
```

## 12. 阶段定义

### 12.1 阶段 0：树莓派盒子本地闭环

目标：

## 14. 隐私、安全与合规要求

商业化落地必须默认把“老人家庭隐私保护”当成产品主约束，而不是上线后补救。

### 14.1 用户授权

- 首次安装必须获得家庭视频感知授权。
- 必须明确告知“本产品不是医疗诊断设备”。
- 必须明确告知上传云端的是事件、截图或短视频证据，不是持续云端录像。
- 家属邀请和协作必须经过家庭空间权限控制。

### 14.2 数据留存

- 原始视频默认不上传云端。
- 事件截图和短视频必须支持可配置留存周期。
- 用户主动删除后，云端和边缘端都要有对应删除机制。
- 运营后台查看媒体必须有授权链路和审计记录。

### 14.3 账号与密钥

- App 侧不保存 RTSP 明文密码。
- 设备必须有独立身份和密钥。
- 日志、导出文件和报表中禁止泄露 RTSP 明文密码、token 和手机号全量信息。
- 管理后台和用户端必须分角色鉴权。

### 14.4 审计与责任边界

- 查看事件、查看媒体、处理误报、修改规则、删除数据都要有审计记录。
- 产品必须把“候选提醒，需要家属确认”的责任边界写清楚。
- 严禁在产品文案中暗示“100% 防跌倒”或“医疗级判断”。

## 17. 2026-07-05 架构边界补充：先本地服务器闭环，再云端化

当前阶段的目标不是让 App 直接连摄像头或盒子，而是先把“边缘盒子 -> App 服务器 -> App/H5”的数据链跑通。

明确边界：

- 摄像头 RTSP、视频解码、抽帧、视觉算法、事件证据包生成留在边缘盒子。
- App/H5 不保存 RTSP 明文，不直接访问摄像头，也不把局域网盒子地址作为日常数据链依赖。首次绑定时允许短暂访问盒子局域网配对入口，绑定完成后仍全部通过云端使用。
- App/H5 面向用户展示设备状态、事件列表、事件详情、截图证据、规则配置和误报反馈。
- 服务器承接设备绑定、设备心跳、配置下发、事件上报、媒体上传、事件查询和审计。
- 第一版服务器优先部署在本地局域网，用来验证完整闭环；闭环稳定后再迁移到云服务器。
- 本地闭环必须保持云端语义：App 写入摄像头配置，服务器生成配置版本，边缘盒子主动拉取并应用配置，再把同步状态、在线状态和错误原因回传给服务器。

这条路线用于保护视频流体验和算法稳定性：

- 实时视频预览和算法推理必须解耦，不能因为 RTMPose / YOLO 推理慢导致用户看到的视频卡死。
- 视觉算法输出应区分“实时预览命中”和“正式事件告警”。预览可以画框、画骨架、显示分数；正式告警必须由规则引擎结合连续帧、持续时间、恢复状态、频控和证据包生成。
- 边缘端默认不上传持续录像，只上传事件截图、短证据和结构化事件数据。

### 17.1 用户态事件与原始事件边界

用户端 App 展示的不是原始日志流，而是当前家庭成员可行动的用户态事件。

产品边界：

- `Event` 原始记录用于审计、排障、模型评估和云端迁移，不应因为前端当前不展示而被删除。
- App 首页、事件页、消息卡片和今日摘要只展示用户态事件。
- 如果摄像头已经恢复在线，历史 `camera_offline` 原始事件可以继续保留，但不应继续作为未处理高危事件提醒家属。
- 今日摘要的未处理数和高危数必须和用户事件列表一致，不能出现“列表为空但首页仍显示高危事件”的矛盾。
- 事件详情只能从用户可见事件进入；内部工程错误、拉流栈信息和设备日志不直接暴露给普通家属。

上云后仍必须保留这条边界：原始事件表负责事实留存，用户态事件视图负责 App 展示和推送决策。

### 17.2 首页关怀信息架构

首页不是功能入口堆叠页，必须围绕“今天要不要关心家里、怎么自然联系老人”组织信息。

首页展示优先级：

1. 家庭状态摘要：只展示盒子、摄像头和待处理事件的真实同步状态。
2. 今日关怀主卡：展示 1:1 图文关怀卡；有生成图片时，图片承载完整图文内容，页面下方只展示生成依据和少量事实点。
3. 最近关怀：横滑展示最近生成的历史关怀卡，旧泛化标题不得继续直接展示。
4. 今日信号：横滑展示天气、日历、回家间隔、老人兴趣话题和设备状态等辅助信号，不再重复今日关怀正文。
5. 位置与回家：在用户授权定位并补充老人城市、上次回家日期后，展示手机与家的距离和回家间隔。

产品文案约束：

- 禁止把首页主卡写成“家里一切平稳”“聊聊家常”“多陪陪家人”等占位句。
- 没有安全事件时，也必须从真实天气、节假日、纪念日、周末、回家间隔或老人兴趣中选一个具体主题。
- 热点和搜索结果只能作为可聊话题候选，不得直接把负面、耸动或广告感标题推给家属。
- 家属通常不在老人身边，关怀动作必须是远程或可安排的动作，例如打电话、发微信、提醒喝水、约定周末回家；不得写成亲手递茶、端水、送到手边、陪在身边等现场照顾动作。

### 17.3 2026-07-09 最小云闭环当前口径

当前产品路径已经从“本地服务器闭环”推进到“最小云闭环验证”。后续判断是否偏离产品方向时，按下面口径执行：

- App/H5、云端服务和树莓派盒子都以云端为同步中心。
- 云端已经承接账号、家庭、老人资料、设备绑定、摄像头配置、规则、关怀卡片、站内消息和通知投递记录。
- 树莓派盒子必须继续主动连云，拉取摄像头和规则配置，并回传在线状态、同步状态和实时帧。
- App/H5 不直接读取盒子局域网 IP，也不直接读取 RTSP 地址。
- 当前实时画面采用盒子到云端的轻量中继验证，不等同于最终视频架构。iOS 阶段可以继续优化当前中继，也可以切换 WebRTC / TURN / MediaMTX 等正式方案。
- 当前通知闭环是 App 内消息和投递记录，不等同于 APNs 真推送。APNs 需要 iOS 真机登记 push token 后再接入。
- 当前 H5 使用局域网发现和自动一次性凭证完成绑定，用户不输入设备码；正式硬件仍建议保留盒身二维码作为 mDNS 失败、换路由器和售后恢复的备用入口。
- 当前公网已迁移到 `https://gohome.ai2shx.club`。iOS 阶段继续补本地网络权限、Bonjour 浏览和 APNs。

当前测试阶段允许在完整数据库备份后清空业务数据，并把真实树莓派盒子恢复为 `claimable`，用于反复验证“注册 -> 家庭 -> 老人资料 -> 绑定盒子 -> 摄像头配置”的完整路径。正式环境不得使用清库方式转移设备，必须走 App 解绑或平台售后流程。

### 17.4 守护能力默认开启原则

- 新用户完成家庭、盒子和摄像头配置后，离线、黑屏、长时间无活动、人形、跌倒、活动状态、烟火和通知能力默认全部开启。
- 默认开启是产品交付状态，不要求普通用户逐项理解算法后再手动打开；用户仍可在“守护设置”中按家庭需要关闭单项能力。
- 跌倒默认开启不等于单帧高分即告警。正式事件必须继续经过站坐基线、下降过程、低位持续、自动床/沙发区域排除和恢复判断。
- 规则默认值必须同时落实在 App 服务、云端下发配置和盒子本地数据库，不能只把前端开关显示为开启。
- 后续规则改为按家庭、盒子或摄像头独立存储时，新建记录仍必须继承全开启默认值。

### 17.5 家庭守护权限与事件闭环

- 守护规则按家庭持久化，并由绑定到该家庭的盒子拉取；不同家庭的算法开关和阈值不得互相影响。
- 只有家庭创建者可以修改守护规则。受邀家庭成员可以查看当前规则，但页面必须保持只读。
- 解绑盒子时必须撤销旧设备 token、移除该家庭摄像头配置并把设备恢复为可认领状态；旧 token 不得被自动归入其他家庭或恢复绑定。
- 正式守护事件必须形成完整数据链：盒子生产算法命中、事件截图、规则证据、云端事件、App 消息和通知投递记录。
- 公开样本回放只用于工程验收，事件必须明确标记为测试，不得冒充老人家庭真实告警。

### 17.6 盒子存储与数据保留

- 盒子普通抽帧、检测结果、规则评估和未升级为正式事件的候选数据默认只保留 24 小时。
- 正式事件、事件截图、事件关联检测链、当前摄像头最新画面、未完成上传和媒体资产不得被自动清理。
- 清理机制必须在盒子后台自动执行，同时在 App“设备管理”展示磁盘、数据库和保留策略。
- 只有家庭创建者可以从 App 发起“立即清理”；家庭成员只查看状态，不能修改盒子维护策略。
- App 清理命令通过云端配置同步下发，盒子分批执行并回传结果，不要求手机直连盒子局域网地址。
- 清理不能改变视觉算法开关、阈值、摄像头配置、家庭事件和老人资料。

### 17.7 单云 HTTPS 产品环境

- 当前产品环境统一部署在腾讯云，正式访问入口为 `https://gohome.ai2shx.club`。
- App 前端、Node 后端、PostgreSQL、媒体文件和 Nginx HTTPS 使用同一云环境，不把另一台云服务器作为长期反向代理依赖。
- 生产环境数据库首次启动不得自动创建“回家管理员”或“默认家庭”；新用户必须从手机号注册开始完成真实引导。
- 盒子主动连接 HTTPS 云端并作为待认领设备出现，用户创建家庭和老人资料后才能认领并配置摄像头。
- 旧阿里云 GoHome 服务已在腾讯云新用户、双摄像头和事件闭环通过后停止并禁用；旧目录和数据库暂时保留，仅用于必要时排障，不再承接产品流量。

### 17.8 局域网安全绑定

- 用户完成注册、家庭和老人资料后，手机与盒子连接同一 Wi-Fi，在 App 点击“搜索并绑定盒子”。
- H5 当前通过 `gohome.local` 进入盒子本地配对入口；iOS App 使用 Bonjour 枚举同一局域网中的 GoHome 盒子。
- App 在云端生成高熵、短时、一次性凭证，由盒子直接向云端兑换设备 token；凭证不展示给用户，也不作为长期设备码。
- 盒子只在启动后的短配对窗口内接受请求。窗口结束后需要重启盒子再配对。
- 云端不得向普通用户列出全局未绑定设备，也不得允许通过序列号、设备 ID 或公开绑定码直接认领。
- 已绑定其他家庭的盒子必须拒绝新家庭抢绑。只有原家庭创建者解绑并撤销旧 token 后，盒子才能重新绑定。
- 局域网只承担首次发现和所有权建立；摄像头配置、规则、视频、事件和关怀数据仍通过 HTTPS 云端同步。

### 17.9 iOS App 壳产品边界

- iOS 第一版使用 SwiftUI + WKWebView 承载现有产品页面，但必须表现为完整 App，而不是要求用户输入网址的浏览器入口。
- App 默认连接腾讯云 HTTPS，使用持久化 Web 数据存储保留登录态；退出登录只能由用户主动操作或服务端撤销会话触发。
- 原生层负责刘海和 Home Indicator 安全区、本地网络权限、Bonjour 盒子发现、定位、系统拨号、微信唤起、深链和 APNs。
- WebView 只允许产品云域名和首次配对所需的 `.local` 地址；不得任意打开第三方网页或把摄像头凭证暴露给其他 App。
- H5 与 iOS 共用账号、家庭、盒子、摄像头、规则、事件和关怀数据，不建立第二套本地业务数据库。
- APNs 只有在真机 token、Apple capability 和服务端投递全部通过后才标记完成；站内消息不能冒充系统推送。

### 17.10 统一视觉感知、姿态时序与云端复核

本节作为下一阶段算法产品口径。iOS 适配暂停在当前可构建状态，先完成算法、数据和云端复核闭环。

#### 17.10.1 产品模块

普通用户和研发管理台都不再按 YOLO、RTMPose、火灾、时序、云端模型拆成多个产品页面。产品只保留一个“视觉感知”能力，内部实现分为三层：

1. 边缘感知引擎：人体、骨架、姿态、场景家具、画面质量和火灾视觉线索。
2. 时序决策引擎：人体轨迹、姿态片段、持续时间、跌倒转变、恢复判断和事件去重。
3. 云端复核与汇总：多模态图片复核、跨摄像头家庭存在状态、事件提醒、分时段摘要和每日汇总。

YOLO、RTMPose、时序状态机和云端模型属于内部技术组件，不是用户需要理解或分别配置的功能模块。

#### 17.10.2 人体存在与姿态片段

- 盒子低频执行人体检测。首版以每秒 1 次为基准，5 秒滑动窗口内至少 4 次可信人体命中，确认 `person_present`，避免单帧误检或一次遮挡重置状态。
- 确认有人后，姿态检测提高到每秒 2-4 次；无人时降频。紧急候选期间允许临时提高采样率。具体频率以两路摄像头实测延迟为准，不写死在产品页面。
- 姿态标签至少覆盖 `standing / sitting / squatting / bending / lying / upper_body / unknown`。当前 `standing_or_sitting` 不能作为目标交付状态。
- 同一人体的同一姿态连续稳定 3 秒，且采样窗口内多数结果一致、平均置信度达标，形成一条 `PostureEpisode`。
- 普通站、坐、蹲、弯腰、躺卧是生活观察片段，不直接生成安全事件。每个片段只保存起止时间、持续时长、置信度、场景关系和少量代表证据，不把每一帧都升级为候选。

#### 17.10.3 跌倒与长时间地面躺卧

跌倒不能等价为单帧躺姿。跌倒候选必须综合姿态因子图：

- 同一人体近期存在站立或坐姿基线。
- 身体中心发生明显向下位移，躯干方向和人体宽高比发生变化。
- 人体进入低位或地面区域并持续未恢复。
- 自动排除床、沙发等正常卧躺区域。
- 结合骨架质量、人体跟踪一致性、运动变化和多帧置信度。

快速跌倒首版目标是在约 4-8 秒内形成边缘候选，不能等待 3 分钟。`lying` 连续超过 3 分钟只在“疑似地面躺卧、非床/沙发、未恢复”时形成 `prolonged_floor_lying` 安全事件。

#### 17.10.4 火灾视觉线索

- 火灾检测继续结合颜色、纹理、区域连通性、画面变化和连续帧，不因单帧暖色物体直接报警。
- 强火灾候选应在连续约 1-2 秒多帧命中后形成边缘紧急事件，并提交云端视觉复核。
- 云端不可用时，达到高置信边缘阈值的火灾事件仍需进入 App，标记为“等待云端复核”，不能静默丢弃。

#### 17.10.5 长时间未见老人

- `long_absence` 由云端按家庭汇总判断，不由单路摄像头直接报警。
- 盒子上报每路摄像头的 `last_person_seen_at`、观察覆盖率、在线状态和人体存在片段；云端维护家庭级 `last_person_seen_at`。
- 默认条件为：所有参与守护的摄像头有效在线、观察覆盖率达标、所有摄像头连续 12 小时没有可信人体数据，并且没有外出、旅行、住院或暂停守护状态。
- 摄像头离线必须生成设备事件，不能被错误解释为“12 小时未见老人”。任一路摄像头看到人，家庭级未见计时立即重置。
- 12 小时是首版默认值，后续可配置为 6、12 或 24 小时。

#### 17.10.6 云端多模态复核

- 当前腾讯云已配置 `Qwen/Qwen3.5-27B`，生产代码已对跌倒、长时间倒地和烟火候选异步发送事件证据图，并以严格 JSON 接收复核结果；模型失败不得阻断边缘事件和 App 提醒。
- 云端复核输入不只是一张图，应包括代表帧或短证据、边缘结构化结果、姿态持续时间、人体轨迹、场景区域和规则命中原因。
- 云端模型返回严格 JSON，至少包含 `person_count / posture / surface / emergency / confidence / reason / suggested_event_type`。
- 单帧模型结果不能单独决定紧急性。快速跌倒和强火灾由边缘端先生成临时紧急事件，云端再确认、降级或标记不确定；普通姿态片段可以等待云端复核后再进入摘要。
- `wan2.7-image` 继续只用于关怀卡生图，不参与监控证据判断。

#### 17.10.7 事件与持续提醒

- 同一事故只创建一条 `SafetyIncident`，不按分钟重复创建事件。
- 事件状态至少包含 `candidate / verifying / confirmed / rejected / acknowledged / resolved`；现有单一 `acknowledged` 布尔值不足以表达复核过程。
- 对已确认或高置信临时紧急事件，默认每 1 分钟产生一条新的提醒投递，直到 App 用户确认收到；提醒记录使用事件 ID 和分钟时间桶幂等。
- App 确认后立即停止提醒。误报必须保留原事件、边缘证据、云端结果和用户反馈，用于后续数据集迭代，不能直接删除。

#### 17.10.8 分时段与每日汇总

- 云端按结构化 `PresenceSession` 和 `PostureEpisode` 汇总上午、下午、夜间和全天活动，不反复扫描原始图片生成生活结论。
- 每日摘要可以描述可验证事实，例如出现时段、活动时长、主要姿态片段、长时间静止和事件处理结果；不得输出医疗诊断。
- 多模态模型只复核代表证据和生成自然语言解释，统计数字由程序计算，避免模型编造时长和次数。

#### 17.10.9 统一视觉感知页面

研发管理台合并为一个页面，包含摄像头切换、唯一真实画面、人体框、人物编号、骨架、当前姿态、自动场景家具、火灾和画面异常状态、最近安全记录、生活观察和运行指标。页面不得提供“选择单个算法”下拉框，不得再显示独立算法演示素材。普通 App 只展示家庭状态、实时画面、事件证据和处理动作，不暴露模型参数或内部算法页面。

#### 17.10.10 数据、隐私与算力

- 数据集以骨架时序、姿态片段和事件证据为主，补充站、坐、蹲、弯腰、正常躺床/沙发、跌倒、跌倒后未恢复、遮挡、低光和多人交叉轨迹。
- 训练集和测试集必须按家庭、摄像头或完整视频序列划分，禁止把相邻帧随机拆分造成数据泄漏。
- 默认只上传事件代表帧、短证据和结构化数据，不持续上传家庭监控录像。
- 现有树莓派算力足以先实现状态与数据架构。AI HAT+ 作为可选加速后端，只有在真实两路摄像头采样率和延迟达不到目标时启用；接入前必须验证目标模型可转换到 Hailo，不能假设插上硬件即可自动加速 RTMPose。

#### 17.10.11 家庭宠物识别边界

- 首版复用 YOLO11 COCO 类别识别猫和狗，不新增模型训练，也不增加独立算法页面。
- 宠物是独立家庭活动状态，不属于人物：不得增加 `person_count`、不得刷新 `last_person_seen_at`、不得进入姿态、跌倒、人物活动片段或长期未见老人计时。
- 研发页在统一真实画面中显示猫狗目标框、置信度和已学习的沙发、床等场景关系；电视画面中的猫狗与电视人物使用同一屏幕内容抑制规则。
- 安全事件证据和云端多模态复核必须携带宠物目标，模型提示明确“宠物在地面、床或沙发上不是人物跌倒证据”。
- 普通 App 后续可显示“未看到老人，但检测到宠物活动”，但宠物活动不能消除 12 小时未见老人的家庭级提醒。
- 本阶段不做宠物身份、健康、情绪、离家或长期未见宠物告警，避免超出家庭老人守护主线。

阶段 12 已于 2026-07-12 完成边缘识别和云端证据链：猫狗与人物、姿态、跌倒和长期未见逻辑已隔离，管理页和持久化数据已支持宠物目标，云端视觉复核上下文已携带宠物信息。下一步使用真实家庭猫狗画面校准置信度和电视/家具误检，再把最新宠物活动摘要加入盒子上报和普通 App 家庭状态。

#### 17.10.12 盒子事件日志与云端状态一致性

- 盒子管理后台必须提供独立事件日志页，但不得建立第二套事故状态。盒子本地 Event 负责离线留证，云端 `SafetyIncident` 负责 App 展示、模型复核、持续提醒和用户确认。
- 只有正式安全或设备事件进入事件日志。实时算法命中、普通姿态、PresenceSession、PostureEpisode 和未晋级候选继续留在视觉研发记录，不得混入用户事故时间线。
- 每条事件按 `edge_event_id` 一一关联盒子本地事件与云端事件，并展示固定链路：盒子触发、事件/截图上传、云端接收、模型或 incident 最终状态。
- 云端状态是最终产品状态，至少展示 `verifying / confirmed / rejected / uncertain / acknowledged / resolved`。盒子断网时显示本地已记录或等待上传，联网后自动合并，不重复创建事件。
- 盒子后台不能代替家属执行“已收到/已处理”，该动作只在 App 中完成并关闭持续提醒。
- 盒子后台允许受保护的“标记算法误报”，该反馈必须先同步云端，再把同一 incident 转为 rejected、归档提醒并保留证据；云端失败时不得只修改本地造成状态分叉。
- 页面允许筛选、刷新、查看本地证据和技术日志，不提供跨模块无意义跳转，也不直接暴露设备令牌、模型 Key 或完整服务端载荷。
