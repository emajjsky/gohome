# 想家了吗 PRD

## 1. 文档目的
这份 PRD 用来定义 `想家了吗` 当前阶段的产品目标、技术栈、MVP 范围和页面职责。

从这一版开始，后续设计和开发默认以本文件为产品基线。

## 2. 产品定义
`想家了吗` 不是一个“看监控”的产品，而是一个“触发回家念头”的产品。

它要解决的问题不是用户不知道父母是否安全，而是用户在长期忙碌之后，逐渐失去对父母状态、情绪和关系距离的真实感知。

一句话定位：

> 不是提醒你看父母，而是提醒你回去陪父母。

## 3. 核心用户
- 在外地工作、回家频率下降的子女
- 对父母有牵挂，但日常感知不足的人
- 能接受科技辅助，但不希望产品过度工具化的人
- 在守护和思念之间，都需要被轻轻提醒的人

## 4. 当前阶段目标
当前阶段不是做完整平台，而是做一个高质量、可展示、可继续迭代的 MVP 原型。

当前目标有 4 个：
- 明确 `想家了吗` 的产品主链路
- 把普通养老守护表达，重构成“情绪触发 + 行动建议”
- 把回忆模式表达，重构成“记忆触发 + 情绪延续”
- 建立一套后续执行时必须遵守的产品和设计基线

## 5. 产品主链路
产品主链路不是“首页 -> 监控 -> 事件”，而是：

1. 先感知
2. 再共情
3. 再触发
4. 再行动

对应页面职责如下：
- 首页：给出“今天为什么该回家”的主触发，以及这周回家安排
- 守护：只提供一眼确认，不把产品做成监控工具
- 牵挂时间线：把当天值得注意的细节整理成简洁证据
- 陪伴日志：展示通话、视频、回家间隔和关系变化
- 我的：设置回家提醒、关系提醒、纪念日和回忆模式开关
- 回忆模式首页：给出“今天适合先想起什么”
- 平行世界：提供情绪出口，让用户继续说没说完的话
- 记忆馆：把照片和记忆整理成可重新打开的人生片段
- 声音：用熟悉的话语触发情绪靠近
- 提醒节奏 / 守护规则：只负责调节提醒强度，不负责讲大道理

## 6. 技术栈
当前项目必须基于现有代码形态继续推进，不在此阶段引入额外前端框架。

### 6.1 当前实现栈
- 页面架构：多页静态 HTML
- 样式方案：`Tailwind CSS 3.4` + 本地 `app.css`
- 构建方式：`tailwindcss` CLI 编译
- 图标方案：本地 `Material Symbols`
- 字体方案：本地 `Noto Sans SC`
- 资源方案：本地图片、本地 SVG、本地字体，不依赖外链资源
- 交互状态：原生 JavaScript + `localStorage`
- 版本管理：Git + GitHub

### 6.2 当前技术栈选择理由
- 足够快：适合持续快速迭代原型
- 足够稳：不引入不必要的复杂工程结构
- 足够可控：所有字体、图标、图片都本地化
- 足够适配当前任务：重点在页面表达，而不是复杂业务系统

### 6.3 当前阶段不做的事
- 不引入 React / Vue / Next.js 等框架重构现有项目
- 不接真实后端服务
- 不接真实监控流媒体能力
- 不做复杂账户体系
- 不做复杂多人协同逻辑

## 7. MVP 功能范围
MVP 必须围绕“触发回家念头”服务，而不是围绕“功能完备”服务。

### 7.1 P0 功能
这些是当前 MVP 必须具备的能力。

#### P0-1 首页主触发
- 每天给出一个最核心的回家理由
- 同时给出立即可执行的动作
- 首页必须具备“今天为什么该回家”的第一感

#### P0-2 回家安排
- 把“想回去”收成一个可以执行的周末安排
- 至少包含距离、出发时间、简短标签和计划感表达

#### P0-3 守护证据
- 用户可以快速看到家里当前是否平稳
- 页面表达必须克制，不能像监控系统
- 只保留必要确认信息

#### P0-4 牵挂时间线
- 用简洁卡片展示当天值得注意的细节
- 允许从列表进入详情
- 标签只做分类，不做煽情表达

#### P0-5 一键行动
- 首页、时间线、陪伴页必须支持直接行动
- 当前 MVP 至少包含：
  - 打电话
  - 视频看看
  - 安排周末回家
  - 加进日历

#### P0-6 陪伴日志
- 展示回家间隔、通话、视频、最近陪伴
- 让用户感知“关系有没有变远”

#### P0-7 回忆模式
- 回忆模式切换可用
- 首页、平行世界、记忆馆、声音、我的导航联动可用
- 切换状态可持久化

#### P0-8 平行世界
- 提供聊天式情绪出口
- 支持视频通话层展示
- 对话界面尽量像真实聊天，而不是功能介绍页

#### P0-9 记忆馆与声音
- 记忆馆支持从“人生片段”切入
- 声音页支持从“今天先听哪一句”切入

### 7.2 P1 功能
这些可以在 MVP 成型后继续增强。

- 家庭协同
- 更细的提醒策略配置
- 更多回忆触发模板
- 更完整的事件详情数据
- 更完整的陪伴趋势分析

## 8. 页面级验收标准
### 8.1 首页
- 不像仪表盘
- 第一屏就能看懂“今天为什么该回家”
- 标题、标签、卡片不过度说明

### 8.2 守护
- 一眼确认即可
- 不出现过强监控感
- 不堆大段解释

### 8.3 牵挂时间线
- 列表规整
- 标签简单直白
- 支持快速进入详情

### 8.4 陪伴日志
- 重点看关系距离，不看堆砌素材
- 数字、标题、模块边界稳定，不顶边不溢出

### 8.5 平行世界
- 页面一进来就是聊天本体
- 不堆顶部介绍卡
- 输入区和通话区逻辑清楚

### 8.6 提醒节奏 / 守护规则
- 直接进入设置项
- 不堆标题区和大段说明
- 保留底部导航

## 9. 文案与设计原则
- 主页面尽量不放冗余顶部说明
- 二级页也保留底部导航，保证能随时返回主链路
- 文案优先短句，不写成一大段解释
- 标签只承担分类作用，不承担煽情表达
- 面板底纹、边框和装饰都要克制，避免原型感太重
- 能直接给动作，就不要先解释一段
- 能一句说清，就不要写三句
- 只有关键节点才需要“事实层 / 情绪层 / 行动层”

## 10. 当前代码与文档绑定规则
从这版开始，后续执行默认遵守以下规则：
- 所有新增功能先判断是否属于本 PRD 的 MVP 范围
- 所有页面调整优先遵循“少说明、少装饰、少边框、少煽情标签”
- 若页面表达与本 PRD 冲突，以本 PRD 为准再回改页面
- 若后续方向发生变化，应先更新 PRD，再继续执行

## 11. 当前版本结论
`想家了吗` 的 MVP 已经不是普通养老守护原型，而是一套围绕“触发回家念头”展开的情绪化产品表达。

下一阶段的重点不再是补很多新功能，而是基于本 PRD，把现有 MVP 做得更完整、更统一、更可展示。
