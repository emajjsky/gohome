# 回家 场景化图文消息补充需求

更新时间：2026-07-01

文档状态：PRD 补充草案

关联文档：

- `想家了吗-PRD.md`
- `想家了吗-Plan.md`
- `想家了吗-Implement.md`

## 1. 文档定位

这份文档补充定义 `回家` 的“场景化图文消息卡片”能力。

它不是单独的新产品方向，而是对现有产品愿景的展开：

> 把老人居家状态、重要日程、天气变化和家属陪伴行为，转化成一张子女愿意打开、愿意行动、能快速处理的图文提醒卡片。

现有 `edge-agent` 负责观察家里，产生检测事实、规则事件和截图证据；本补充需求定义 App 和云端如何把这些事实结合老人资料、日历、天气、通话/回家记录，生成更有温度、更可行动的消息。

### 1.1 与主文档的对齐原则

这份补充文档必须服从当前主 `PRD / Plan / Implement` 的统一口径：

- 阶段命名统一使用 `阶段 0 / 阶段 1 / 阶段 2`。
- `阶段 0` 允许本地 mock 或本地接口验证图文消息展示，但这不代表正式云端消息能力已完成。
- 正式云端核心消息对象统一使用 `MessageCandidate`。
- `ReminderCard` 在这份文档里只表示 App/H5 的卡片渲染形态，不再作为独立核心领域对象。
- 推送结果、日志和审计统一复用主文档已定义的 `NotificationReceipt`、`DeviceLog`、`AuditLog`。
- 正式接口统一收口到 `/api/v1`，并与主文档的 `message-service`、`log-service`、`notification-service` 保持一致。

## 2. 产品目标

### 2.1 核心目标

场景化图文消息卡片要解决四件事：

1. 让安全提醒不只是报警，而是能解释“发生了什么、为什么提醒、现在该做什么”。
2. 让陪伴提醒不依赖用户主动打开 App，而是由系统在合适时间主动触达。
3. 让日历、天气、老人资料和视觉检测形成组合价值，而不是各自孤立。
4. 让用户端 App 从“看监控”升级为“家庭牵挂助手”。

### 2.2 用户感知

用户收到的不是冷冰冰的系统通知，而是类似下面这样的图文卡片：

```text
明天是妈妈生日

她喜欢桂花糕，也喜欢你回家吃顿饭。
建议今天准备一份小礼物，明晚回家陪她过生日。

依据：家庭日历 + 妈妈资料
动作：准备礼物 / 打电话 / 安排回家
```

### 2.3 产品边界

必须明确：

- 本能力不是医疗诊断，不对疾病做预测。
- 本能力不是替用户做道德评判，不使用羞辱式文案。
- 本能力不是持续监控老人隐私，默认只基于事件、摘要和必要证据生成提醒。
- 安全告警优先使用真实截图或短视频证据，不使用 AI 图代替事实证据。
- AI 文生图只用于情感类、生活类、节日类卡片，不用于跌倒、黑屏、离线等需要证据的告警。

## 3. 能力总览

场景化图文消息卡片由五类能力组成：

```text
老人资料档案
+ 日历和重要日期
+ 天气和外部环境
+ 视觉检测与事件
+ 家属行为记录
-> Reminder Engine / message-service
-> MessageCandidate
-> 图文消息卡片渲染
-> App 首页 / 推送 / 事件页 / 家属协作
```

### 3.1 老人资料档案

用于让提醒具备个性化和现实可执行性。

必填字段：

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| elder_id | 老人 ID | `elder_001` |
| display_name | App 展示称呼 | 妈妈 |
| relationship | 与用户关系 | 母亲 |
| city | 常住城市 | 苏州 |
| birthday | 公历生日 | `1958-08-12` |
| lunar_birthday | 农历生日，可选 | 七月初一 |
| living_status | 居住状态 | 独居 |
| primary_room | 主要活动空间 | 客厅 |

推荐字段：

| 字段 | 说明 | 示例 |
| --- | --- | --- |
| likes | 喜好 | 桂花糕、鲜花、听评弹、散步 |
| dislikes | 不喜欢或禁忌 | 太甜、太辣 |
| diet_notes | 饮食注意 | 少盐、低糖 |
| health_conditions | 基础疾病标签 | 高血压、糖尿病、膝关节不适 |
| medication_notes | 用药提醒备注 | 早晚饭后服药 |
| routine | 作息习惯 | 7 点起床，12 点午休，21 点睡觉 |
| emergency_contacts | 紧急联系人 | 子女、邻居、社区医生 |
| home_address | 常住地址粗粒度 | 市区/小区，不默认展示门牌 |

隐私要求：

- 基础疾病、用药信息属于敏感信息，默认只在授权家庭成员之间可见。
- 消息卡片中不直接暴露完整病史，只转化为温和提醒。
- 卡片文案应避免“你妈妈有病所以不能出门”这类表达。

### 3.2 日历和重要日期

用于生成节日、生日、纪念日、体检、回家计划等提醒。

数据来源：

- 用户手动录入
- 手机系统日历授权
- App 内家庭日历
- 后续云端家庭空间共享日历

事件类型：

| 类型 | 示例 | 提醒目标 |
| --- | --- | --- |
| 生日 | 妈妈生日、爸爸生日 | 准备礼物、回家吃饭、打电话 |
| 节日 | 春节、中秋、重阳节、母亲节 | 安排陪伴、提前购票、问候 |
| 体检 | 社区体检、医院复诊 | 提醒陪同、带证件、准备报告 |
| 用药/护理 | 定期取药、复查 | 提醒关注健康安排 |
| 家庭计划 | 周末回家、一起吃饭 | 把建议变成计划 |

### 3.3 天气和外部环境

用于生成生活类关怀提醒。

第一阶段需要接入：

- 城市天气
- 未来 24 小时天气
- 未来 7 天天气
- 温度变化
- 降雨、雷暴、大风、寒潮、高温
- 空气质量，可选

天气提醒类型：

| 类型 | 条件示例 | 卡片建议 |
| --- | --- | --- |
| 雷暴雨 | 今日雷暴雨或暴雨预警 | 提醒妈妈少出门，关窗，带伞 |
| 降温 | 24 小时降温超过 6 摄氏度 | 提醒加衣、关窗、注意保暖 |
| 高温 | 最高温超过 35 摄氏度 | 提醒减少外出、补水、开空调 |
| 大风 | 大风预警 | 提醒收衣服、少去阳台 |
| 空气差 | 空气质量差 | 提醒减少户外活动 |

### 3.4 视觉检测与事件

继续沿用当前 `edge-agent` 的定位：

- 摄像头接入
- 抽帧
- 画面质量检测
- 人形检测
- 运动/变化检测
- 疑似跌倒候选
- 规则事件
- 截图证据

卡片生成时，视觉侧只提供事实和证据，不直接输出情绪化结论。

示例：

```json
{
  "source": "vision_event",
  "event_type": "long_sitting",
  "room": "客厅",
  "duration_minutes": 82,
  "confidence": 0.72,
  "snapshot_url": "/media/snapshot/xxx.jpg",
  "occurred_at": "2026-07-01T15:20:00+08:00"
}
```

### 3.5 家属行为记录

用于判断“是否需要提醒陪伴”。

建议记录：

| 数据 | 来源 | 用途 |
| --- | --- | --- |
| 最近一次回家 | 用户手动确认、定位授权、日历计划 | 久未回家提醒 |
| 最近一次通话 | 用户手动记录、通讯录/通话授权可选 | 久未联系提醒 |
| 最近一次事件处理 | App 事件处理记录 | 判断家属是否已响应 |
| 家庭成员分工 | 家庭空间 | 多子女协作提醒 |
| 已忽略提醒 | 用户反馈 | 降低打扰 |

注意：

- 通话记录和定位属于高敏感权限，MVP 可以先手动记录，不强制读取系统权限。
- 久未回家提醒要避免道德绑架，应提供可执行建议。

## 4. 消息卡片模型

本节和主文档的消息口径保持一致：

- 云端正式对象：`MessageCandidate`
- App/H5 展示对象：图文消息卡片渲染结果

也就是说，产品上用户看到的是“卡片”，但系统契约层优先围绕 `MessageCandidate` 设计。

### 4.1 卡片场景类型

| 场景类型 | 说明 | 是否需要真实证据 | 是否允许 AI 图 |
| --- | --- | --- | --- |
| safety_alert | 安全告警，如疑似跌倒、离线、黑屏 | 是 | 否 |
| care_suggestion | 关怀建议，如久未回家、久未通话 | 否 | 是 |
| life_reminder | 生活提醒，如天气、降温、高温 | 否 | 是 |
| calendar_reminder | 日期提醒，如生日、节日、体检 | 否 | 是 |
| quiet_status | 温和状态摘要，如今天家里很安静 | 可选 | 是 |
| device_notice | 设备提醒，如摄像头离线、盒子断网 | 是，至少状态证据 | 否 |

### 4.2 卡片字段

第一版建议结构应兼容主文档里的 `MessageCandidate`：

```json
{
  "message_id": "msg_20260701_001",
  "family_id": "family_001",
  "device_id": "dev_xxx",
  "elder_id": "elder_001",
  "message_type": "gohome",
  "priority": "warm",
  "title": "明天是妈妈生日",
  "subtitle": "她喜欢桂花糕，也喜欢你回家吃顿饭。",
  "body": "建议今天准备一份小礼物，明晚回家陪她过生日。",
  "facts": [
    "生日：7 月 2 日",
    "喜好：桂花糕、鲜花",
    "你上次回家：5 天前"
  ],
  "image": {
    "mode": "generated",
    "url": "/cards/images/msg_20260701_001.png",
    "prompt": "温暖的家庭餐桌，生日蛋糕，桂花糕，一盏暖灯，真实生活摄影风格，不出现具体人脸"
  },
  "evidence": {
    "snapshot_url": null,
    "event_id": null
  },
  "actions": [
    {
      "key": "call",
      "label": "打电话"
    },
    {
      "key": "prepare_gift",
      "label": "准备礼物"
    },
    {
      "key": "schedule_visit",
      "label": "安排回家"
    }
  ],
  "source": [
    "elder_profile",
    "calendar",
    "visit_record"
  ],
  "source_event_ids": [],
  "source_media_ids": [],
  "generated_by": "calendar_rule_v1",
  "created_at": "2026-07-01T08:00:00+08:00",
  "expires_at": "2026-07-02T23:59:59+08:00",
  "status": "open"
}
```

### 4.3 优先级

| 优先级 | 用途 | 推送策略 |
| --- | --- | --- |
| critical | 疑似跌倒、长时间无响应、紧急设备失效 | 立即推送，可重复升级 |
| warning | 黑屏、离线、长时间无活动、异常夜间活动 | 尽快推送，限制频次 |
| warm | 生日、久未回家、久未通话、天气关怀 | 合适时间推送 |
| summary | 今日状态摘要、平稳提醒 | App 内展示为主，少推送 |

补充对齐：

- `message_type` 在正式云端对象中建议进一步收口到主文档定义的 `alert / explain / accompany / gohome` 四类。
- 本文中的 `safety_alert / care_suggestion / life_reminder / calendar_reminder / quiet_status / device_notice` 更适合作为场景标签或卡片模板类型，而不是独立于主对象之外的正式顶层枚举。

## 5. 重点展示场景

下面场景优先用于产品 Demo、App 首页、推送卡片和路演展示。

### 5.1 生日提醒

触发条件：

- 明天或今天是老人生日。
- 老人资料中有喜好字段。
- 家属最近 3 到 14 天未回家，优先级提高。

卡片示例：

```text
明天是妈妈生日

她喜欢桂花糕，也喜欢你回家吃顿饭。
建议今天准备一份小礼物，明晚回家陪她过生日。

依据：家庭日历 + 妈妈喜好 + 回家记录
动作：准备礼物 / 打电话 / 安排回家
```

图像策略：

- 允许 AI 生成温暖家庭餐桌、生日蛋糕、礼物图。
- 不生成老人真实人脸。
- 不生成过度煽情、孤独等待的画面。

### 5.2 久未回家提醒

触发条件：

- 用户超过设定天数没有标记回家。
- 默认阈值：5 天。
- 如果临近周末或节日，可提高提醒权重。

卡片示例：

```text
你已经 5 天没有回家了

妈妈这几天状态平稳，但家里一直很安静。
这周可以安排一次回家，哪怕只是陪她吃顿晚饭。

依据：回家记录 + 家里状态摘要
动作：安排周末回家 / 先打电话 / 暂不提醒
```

文案要求：

- 不写“你太久没回家了”。
- 不写“妈妈很孤独”作为确定事实，除非用户手动记录。
- 用“建议”“可以”“先”降低压力。

### 5.3 久未通话提醒

触发条件：

- 用户超过设定天数没有通话或手动记录联系。
- 默认阈值：3 天。
- 老人当天活动偏少或天气恶劣时，提醒权重提高。

卡片示例：

```text
今天适合给妈妈打个电话

你们已经 3 天没有通话了。
今天下午她在客厅坐了比较久，可以先问问她吃饭了吗。

依据：联系记录 + 今日活动摘要
动作：现在打电话 / 发消息 / 晚点提醒
```

### 5.4 长时间静坐 / 发呆

触发条件：

- 客厅或常用房间检测到人形存在。
- 连续较长时间位置变化很少。
- 没有跌倒候选，未达到紧急等级。

卡片示例：

```text
妈妈今天在客厅坐了很久

不是紧急异常，但下午活动比平时少。
建议先打个电话，问问她今天精神怎么样。

依据：客厅画面摘要 + 活动变化
动作：打电话 / 查看截图 / 标记已确认
```

证据策略：

- 可提供最新截图，但默认不在推送通知外露敏感画面。
- App 内点开后再看证据。
- 卡片主图可以使用柔和插画或抽象客厅图，不替代真实截图。

### 5.5 疑似跌倒告警

触发条件：

- 视觉模型输出疑似跌倒候选。
- 规则层确认持续时间、姿态、位置等条件达到阈值。

卡片示例：

```text
客厅出现疑似跌倒，请立即确认

系统检测到异常姿态，已保存截图证据。
请先查看画面并联系妈妈，如无法联系，请通知邻居或紧急联系人。

依据：视觉检测 + 规则事件 + 截图证据
动作：查看证据 / 打电话 / 通知家人 / 标记误报
```

要求：

- 必须使用真实截图或短视频证据。
- 不使用 AI 生成图。
- 必须提供“误报”反馈入口，用于后续算法评估。
- 支持升级通知策略，例如 5 分钟未处理时通知其他家庭成员。

### 5.6 雷暴雨提醒

触发条件：

- 老人所在城市今日有雷暴雨、暴雨或强对流预警。
- 如果老人有外出习惯或当天有体检/买菜计划，提醒权重提高。

卡片示例：

```text
今天苏州有雷暴雨

建议提醒妈妈少出门，关好窗户。
如果她要买菜，可以帮她叫个配送，或者晚点再出门。

依据：天气预报 + 妈妈常住城市
动作：发消息 / 打电话 / 帮她叫配送
```

图像策略：

- 可使用天气模板图或 AI 生成雨天窗边暖灯图。
- 不制造灾难感。

### 5.7 降温 / 高温提醒

触发条件：

- 24 小时降温超过阈值。
- 最高温超过高温阈值。
- 老人有高血压、心脏病、关节问题等基础疾病标签时，提醒权重提高。

卡片示例：

```text
今晚会明显降温

妈妈所在城市今晚最低温 3°C。
建议提醒她加衣服，睡前关好窗户。

依据：天气预报 + 常住城市
动作：发消息 / 打电话 / 明早再提醒
```

健康边界：

- 可以说“注意保暖”“减少外出”。
- 不说“会诱发某疾病”。
- 不做医学判断。

### 5.8 体检 / 复诊提醒

触发条件：

- 家庭日历中存在体检、复诊、取药事件。
- 事件前 1 天、当天早上提醒。

卡片示例：

```text
明天妈妈要去社区体检

建议今晚确认一下证件和医保卡。
如果你有时间，可以陪她一起去。

依据：家庭日历 + 健康安排
动作：设置提醒 / 打电话确认 / 安排陪同
```

### 5.9 早餐 / 作息异常

触发条件：

- 老人档案中有常规起床或早餐时间。
- 当天在常规时间段内未检测到正常活动。
- 未达到紧急告警等级。

卡片示例：

```text
今天早上的活动比平时少

平时这个时间妈妈已经在客厅或厨房活动。
今天还没有明显变化，可以晚点再观察，或先发条消息问问。

依据：作息习惯 + 活动检测
动作：继续观察 / 发消息 / 打电话
```

要求：

- 避免过早制造焦虑。
- 默认先作为 App 内提醒，连续异常再推送。

### 5.10 夜间异常活动

触发条件：

- 凌晨时间段检测到活动。
- 与老人平时作息不一致。
- 不符合跌倒或安全告警时，作为次日提醒。

卡片示例：

```text
昨晚客厅有一次夜间活动

凌晨 2:18 检测到客厅短暂活动。
不一定是异常，建议今天早上问问妈妈昨晚睡得怎么样。

依据：夜间活动检测 + 作息习惯
动作：打电话 / 标记正常 / 调整提醒
```

### 5.11 摄像头黑屏 / 离线

触发条件：

- 摄像头离线。
- 画面黑屏或被遮挡超过阈值。

卡片示例：

```text
客厅摄像头暂时不可用

当前无法确认家里状态。
请检查摄像头电源、网络，或联系家里人看一下设备。

依据：设备状态 + 画面质量检测
动作：查看设备 / 联系家人 / 标记已处理
```

要求：

- 设备类卡片要明确“当前无法确认状态”，不要误写成老人异常。
- 不使用 AI 图。

## 6. Reminder Engine 规则

### 6.1 输入

Reminder Engine 第一版输入：

| 输入 | 来源 | 阶段 |
| --- | --- | --- |
| ElderProfile | 老人资料档案 | 阶段 0 可本地 mock，阶段 2 入云端 |
| CalendarEvent | 家庭日历/用户录入 | 阶段 0 本地录入，阶段 2 云端 |
| WeatherForecast | 天气 API | 阶段 0 可直连第三方，阶段 2 经云端 |
| VisionEvent | edge-agent 事件 | 已有基础 |
| ContactRecord | 通话/消息/手动记录 | 阶段 0 手动记录 |
| VisitRecord | 回家记录/计划 | 阶段 0 手动记录 |
| UserPreference | 提醒偏好 | 阶段 2 |

### 6.2 输出

输出统一以 `MessageCandidate` 为云端核心对象，再在 App 侧渲染成卡片：

```text
MessageCandidate
-> 卡片渲染模型
-> App 首页卡片
-> 推送通知卡片
-> 事件/消息列表
-> 家庭成员协作状态
```

### 6.3 冲突和去重

规则：

- 同一老人、同一主题、同一天最多主动推送 1 次。
- `critical` 安全告警不受普通频控限制。
- `warning` 设备/安全事件可以重复，但需要升级策略。
- `warm` 关怀提醒不在深夜推送。
- 用户已标记“不再提醒”后，当前主题进入冷却期。
- 如果多个场景同时触发，优先级为：
  1. 疑似跌倒 / 紧急安全
  2. 摄像头离线 / 黑屏
  3. 长时间无活动 / 夜间异常
  4. 天气风险
  5. 生日 / 日历
  6. 久未回家 / 久未通话
  7. 今日摘要

### 6.4 推送时间

建议默认时间：

| 卡片类型 | 推送时间 |
| --- | --- |
| 安全告警 | 立即 |
| 设备异常 | 立即或 5 分钟确认后 |
| 天气提醒 | 当天 7:30 到 9:00 |
| 生日/节日 | 前一天 20:00 或当天 8:00 |
| 久未通话 | 18:00 到 20:30 |
| 久未回家 | 周四/周五 18:00 到 20:30 |
| 今日摘要 | 晚上 20:00 到 21:30 |

## 7. AI 文生图策略

### 7.1 使用范围

允许使用 AI 图：

- 生日
- 节日
- 久未回家
- 久未通话
- 天气生活提醒
- 今日温和摘要

不允许使用 AI 图：

- 疑似跌倒
- 长时间无响应
- 黑屏 / 遮挡 / 离线
- 任何需要证明事实的安全事件

### 7.2 图像风格

推荐方向：

- 温暖、真实生活摄影感
- 家庭灯光、餐桌、窗边、雨天、礼物、电话、回家路
- 不出现具体老人真实面孔
- 不制造过度孤独、病弱、哭泣、灾难化画面

### 7.3 Prompt 生成原则

Prompt 不直接包含隐私字段：

- 不包含老人真实姓名。
- 不包含具体地址。
- 不包含具体疾病名称，除非是内部不可见的生成参考，且不进入图像文案。
- 不生成和真实老人高度相似的人脸。

示例：

```text
温暖的家庭餐桌，生日蛋糕和一盒中式点心，窗外傍晚光线，真实生活摄影风格，柔和自然光，不出现具体人脸，不出现文字
```

## 8. App 展示要求

### 8.1 首页

首页应展示：

- 今日最重要的一张提醒卡片
- 老人当前状态摘要
- 最近一条安全事件
- 快捷动作：打电话、发消息、安排回家、查看事件

首页不应展示：

- 一堆技术指标
- 原始算法结果
- 多个重复跳转入口

### 8.2 消息列表

消息列表按时间展示所有卡片：

- 未处理
- 已确认
- 已忽略
- 已完成动作

过滤项：

- 安全
- 陪伴
- 天气
- 日历
- 设备

### 8.3 卡片详情

详情页应包含：

- 卡片标题
- 主图或证据图
- 事实依据
- 系统判断
- 建议动作
- 家庭成员处理状态
- 反馈入口：有用、无用、误报、不再提醒

### 8.4 推送通知

推送通知分两层：

1. 系统通知标题和一句话摘要。
2. 点开后进入 App 图文卡片详情。

示例：

```text
明天是妈妈生日
建议准备一份小礼物，明晚回家陪她吃饭。
```

安全告警示例：

```text
客厅出现疑似跌倒，请立即确认
已保存截图证据，请尽快联系妈妈。
```

## 10. API 补充范围

### 10.1 阶段 0 本地验证接口

阶段 0 可以先在本地或 mock 数据中完成：

- `GET /api/elder-profile`
- `PUT /api/elder-profile`
- `GET /api/calendar-events`
- `POST /api/calendar-events`
- `GET /api/weather/today`
- `GET /api/reminder-cards`
- `POST /api/reminder-cards/generate`
- `PATCH /api/reminder-cards/{id}`

### 10.2 正式云端接口

进入正式云端后，接口必须和主文档现有 `/api/v1` 口径对齐。

其中分两层：

1. 主文档已经排进消息主链的接口
2. 本补充需求新增的老人资料 / 日历 / 天气域接口

#### A. 已与主文档对齐的消息主链接口

- `POST /api/v1/internal/messages/generate`
- `GET /api/v1/app/messages`
- `GET /api/v1/app/messages/{message_id}`
- `PATCH /api/v1/app/messages/{message_id}`

#### B. 本补充需求新增，但不应抢跑在最小云第一批之前的域接口

下面这些接口属于第二批消息域输入，应在最小云主链稳定后再进入实现：

- `GET /api/v1/families/{family_id}/elders/{elder_id}/profile`
- `PUT /api/v1/families/{family_id}/elders/{elder_id}/profile`
- `GET /api/v1/families/{family_id}/calendar-events`
- `POST /api/v1/families/{family_id}/calendar-events`
- `GET /api/v1/families/{family_id}/weather-signals`

对齐约束：

- 不再新增独立的 `/api/v1/families/{family_id}/reminder-cards` 正式主接口。
- 用户端统一通过 `/api/v1/app/messages` 读取消息，再由前端渲染成场景化图文卡片。
- 老人资料、日历、天气信号属于消息生成输入域，不替代消息主对象。

### 10.3 第三方 API

需要接入：

- 天气 API：高德、和风天气、OpenWeather 等择一。
- 日历：第一版先 App 内录入，后续再接手机系统日历授权。
- 文生图：第一版可以云端异步生成，失败时回退模板图。
- 推送：正式阶段走 APNs / 厂商推送，本地阶段可继续临时通道。

## 11. MVP 优先级

### 11.1 第一阶段：可演示

目标：做出能展示产品价值的卡片，不追求所有外部 API 完整接入。

这里的“第一阶段”对应主文档 `阶段 0` 的演示验证，不等于正式云端消息能力上线。

必须完成：

1. 老人资料档案表单。
2. 生日、喜好、城市、基础健康备注字段。
3. 手动录入重要日期。
4. 手动或 mock 天气数据。
5. 基于现有事件生成安全类卡片。
6. 基于资料 + 日期 + 天气生成生活类卡片。
7. App 首页展示一张主卡片。
8. 消息列表展示多张历史卡片。

优先场景：

- 生日提醒
- 久未回家
- 久未通话
- 雷暴雨提醒
- 长时间静坐
- 疑似跌倒
- 摄像头离线

### 11.2 第二阶段：真实接入

必须完成：

1. 天气 API 真实接入。
2. 家庭日历云端存储。
3. Reminder Engine 去重和频控。
4. 图文卡片推送。
5. 文生图异步生成和模板回退。
6. 家庭成员处理状态同步。

这里的“第二阶段”对应主文档 `阶段 2` 中消息能力的第二批接入，应晚于：

- 设备注册、绑定、心跳
- 事件上报与查询
- 媒体上传与授权访问
- 最小消息主链 `/api/v1/app/messages`

### 11.3 第三阶段：个性化和商业化

后续能力：

- 多老人家庭。
- 多子女协作。
- 节日礼物推荐。
- 附近配送、鲜花、蛋糕、买药等服务入口。
- 告警升级通知。
- 误报反馈驱动规则优化。
- 运营后台查看卡片触达率、点击率、处理率。

## 12. 验收标准

### 12.1 产品验收

一条合格卡片必须满足：

- 用户能在 3 秒内看懂发生了什么。
- 用户能看懂提醒依据。
- 用户有明确下一步动作。
- 文案不制造羞辱感或恐慌感。
- 安全事件有真实证据或明确状态依据。
- 情感/生活类卡片不冒充事实证据。

### 12.2 技术验收

MVP 验收：

- 能保存老人资料。
- 能保存至少 3 类重要日期。
- 能读取或 mock 天气。
- 能从现有事件生成安全卡片。
- 能生成至少 7 类场景卡片。
- 能在 App 首页和消息列表展示。
- 能对卡片执行已读、已处理、忽略、误报反馈。
- 卡片生成失败时不影响安全告警推送。

### 12.3 隐私验收

必须满足：

- 推送锁屏文案不暴露敏感疾病和家庭地址。
- 疑似跌倒等安全截图默认不直接显示在系统通知中。
- 文生图不生成老人真实肖像。
- 基础疾病、用药、地址只在授权家庭成员内可见。
- 用户可以关闭生活类提醒，仅保留安全类提醒。

## 13. 与现有系统的关系

现有 `edge-agent`：

- 继续负责本地摄像头、检测、规则和事件。
- 不负责生成所有亲情文案。
- 不负责长期保存家庭日历和老人完整资料。

未来云端：

- 负责老人资料、家庭日历、天气信号、卡片生成、推送和家庭成员状态。
- 负责把 edge-agent 上报的事件转成 App 可展示的消息。

App / H5：

- 负责展示卡片、处理动作、反馈结果。
- 不直接访问摄像头 RTSP。
- 不直接承担算法判断。

## 14. 近期建议执行顺序

建议按下面顺序进入实现：

1. `阶段 0` 先用本地 mock 数据验证：
   - 老人资料输入
   - 生日 / 久未回家 / 雷暴雨 / 长时间静坐 / 疑似跌倒等卡片展示
2. 卡片底层统一按 `MessageCandidate` 设计，不再单独扩 `ReminderCard` 作为核心领域对象。
3. 在 App 首页做一张主图文卡片，并在消息列表里验证多卡片展示。
4. 先完成主文档里的最小云主链：
   - 设备身份
   - 事件
   - 媒体
   - 播放会话
5. 再接主文档已定义的消息主链：
   - `POST /api/v1/internal/messages/generate`
   - `GET /api/v1/app/messages`
   - `GET /api/v1/app/messages/{message_id}`
   - `PATCH /api/v1/app/messages/{message_id}`
6. 最后再进入第二批消息域输入：
   - 老人资料
   - 家庭日历
   - 天气信号
   - 文生图和模板回退
   - 正式图文卡片推送

## 15. 当前结论

场景化图文消息卡片是 `回家` 从“安全监控工具”走向“亲情陪伴产品”的关键能力。

它的核心不是多发通知，而是把下面这些数据组合成一个更像人的提醒：

- 老人是谁
- 她喜欢什么
- 今天外部环境如何
- 家里状态是否平稳
- 子女多久没联系
- 现在最适合做什么

第一版不要追求复杂 AI。先把生日、天气、久未回家、久未通话、长时间静坐、疑似跌倒这几类场景跑通，就足以支撑产品展示和下一步 App 设计。
